### Backup Strategies Overview  
The video begins by emphasising the critical importance of **backing up important data** and introduces different approaches to deciding **when and how to back up data**. It outlines that data which changes frequently demands more nuanced backup strategies than static data.  

- **Full backup:** A complete copy of all data is made regardless of whether files have changed.  
- For **rarely changing files** (e.g., operating system configuration), full backups are **inefficient** because they waste storage and bandwidth by duplicating unchanged data.

### Differential Backup Explained  
The video describes **differential backups** as an efficient alternative:  
- It **backs up only files that have changed or been created since the last full backup**.  
- This saves storage space and reduces backup time by avoiding duplication of unmodified files.  

However, it stresses that full backups must still be taken periodically because:  
- Over time, differential backups accumulate many changed files, increasing storage requirements.  
- A **best practice** is to perform **infrequent full backups combined with frequent differential backups**.  

 
### Backup Scheduling and Data Loss Considerations  
The frequency of full backups is dictated by how far back you want to be able to recover data. An example schedule:  
- Full backups once per week  
- Differential backups daily  

This schedule results in a **worst-case data loss window of approximately 24 hours**, which is generally acceptable for many use cases.  

 
### Incremental Backups and Their Efficiency  
The video introduces **incremental backups** as an even more efficient method than differential backups:  
- Incremental backups store only the **data changes within files** since the last incremental backup, not just changed files themselves.  
- This approach minimises disk space and backup time further than differential backups.  

However, incremental backups come with important trade-offs:  
- To fully restore data, **all incremental backups since the last full backup are required**.  
- If any incremental backup is *missing or corrupt*, data recovery may be limited to the last full backup only.  
- **Recovery time can be longer**, as the restoration process must integrate the full backup with each subsequent incremental backup.  
- Large or frequently changing files increase the time needed for recovery.  

 
### Backup Compression and Its Trade-offs  
The video explains how **file compression** can reduce backup storage needs:  
- Backup files and folder structures are archived into a single package.  
- Compression uses complex algorithms to store data in less space.  
- **Compression effectiveness varies by data type**; some data compresses well, others do not.  

Important caveats include:  
- Compressed backups require **decompression before restoration**, which can be time-consuming and resource-intensive depending on backup size.  

 
### Storage Considerations for Backup Data  
The video discusses options for storing backup data:  
- Commercial **Network Attached Storage (NAS)** devices or fast servers with large disk capacity can be used.  
- As backup data grows, a single large disk (e.g., 10 terabytes) may fill quickly.  
- The availability of disks larger than 10 terabytes is *Not specified*.  
- Critical risk: a **single disk failure can result in total data loss** if backups are stored on one disk only.

### RAID Arrays for Storage Reliability and Capacity  
To address storage growth and failure risk, the video introduces **RAID (Redundant Array of Independent Disks)**:  
- RAID combines multiple physical disks into one large virtual disk.  
- Different **RAID levels** offer various balances of **performance, capacity, and reliability**.  
- RAID arrays provide:  
  - Increased total storage capacity  
  - Fault tolerance to minimise data loss from disk failures  
  - Flexibility to expand capacity in the future  

**Key clarification:**  
- RAID is a **storage solution, not a backup solution**.  
- It provides **hardware redundancy** but does not protect against:  
  - Accidental file deletion  
  - Data corruption (e.g., malware)  

The video stresses this distinction multiple times to prevent misunderstanding.

### Summary Table: Backup Types and Characteristics

| Backup Type       | Description                                      | Storage Efficiency        | Recovery Complexity                  | Risk Factors                         |
|-------------------|------------------------------------------------|---------------------------|------------------------------------|------------------------------------|
| Full Backup       | Copies all data regardless of changes          | Inefficient for large data | Simple and fast recovery            | High storage use                   |
| Differential Backup | Copies files changed since last full backup   | More efficient            | Moderate recovery time              | Growing storage as changes accumulate |
| Incremental Backup | Copies only data changes since last incremental | Most efficient            | Longest recovery time, needs all increments | Risk of data loss if any increment missing |

---

### Key Insights  
- **Regular full backups are essential** to avoid unmanageable storage growth and ensure reliable recovery.  
- **Differential backups optimise for changes since last full backup**, reducing space but requiring regular full backups.  
- **Incremental backups are most space- and time-efficient**, but recovery is slower and more fragile.  
- **Backup compression saves space but adds decompression overhead during restoration.**  
- **RAID arrays improve storage reliability and capacity but do not replace backups.**  
- Always maintain **separate backups in addition to RAID** to protect against accidental deletion and corruption.  

---

### Conclusions  
Backing up data effectively requires balancing **backup frequency, storage efficiency, and recovery time**. Combining **infrequent full backups with frequent differential or incremental backups** is a common best practice. Compression and RAID can aid in managing storage, but **do not substitute for proper backup strategies**. Understanding these trade-offs is critical for designing a robust data protection plan.
