# Notes
  
This lesson introduces **Active Directory (AD)**, the native directory service for Microsoft Windows, first released with Windows Server 2000. AD is integral to centrally managing networks of Windows computers. It supports the **LDAP protocol**, enabling interoperability with other operating systems such as Linux and OSX, thus facilitating cross-platform directory services.

Key points:  
- Active Directory is widely used in Windows-based organisational networks.  
- AD supports LDAP, allowing it to integrate with non-Windows systems.  
- Its primary function extends beyond directory services to centralised management of networked computers.

Active Directory’s functionality is multifaceted: it not only provides **centralised authentication** and directory services but also acts as the central repository for **Group Policy Objects (GPOs)**. GPOs are essential for managing and enforcing configuration policies across Windows servers and client machines.

Highlights:  
- AD manages both authentication and configuration via GPOs.  
- GPOs enable system administrators to configure Windows machines consistently and centrally.

Important notes on AD structure:  
- Active Directory is **hierarchical** and object-based.  
- Objects include users, computers, groups, and containers.  
- Containers can be either simple containers or **Organisational Units (OUs)**.  
- Unlike ordinary containers, OUs can nest other OUs, allowing for complex organisational structures.

If you look at the logical shape of a domain, it looks like a tree.

The name, even makes sense.

The hierarchical nature of AD is illustrated using the ADAC’s **tree view**. This visualises the directory’s layers and objects, emphasising the layered structure of containers and OUs. Key clarifications:  
- Ordinary containers do not contain other containers.  
- OUs are flexible containers that can nest other OUs, supporting a hierarchical organisation.

 
The root node in AD’s hierarchy is the **domain**, which has both a **short name** (e.g., *example*) and a **DNS name** (e.g., *example.com*). Computers and other objects within the domain inherit DNS names within the domain’s DNS zone.

Additional structural detail:  
- Above the domain level is the **forest**, which may contain one or more domains.  
- Domains within a forest can share accounts and resources.  
- The example environment discussed has a single domain, *example.com*, within its forest.

Common containers within a domain include:  
- **Computers container**: where new computer accounts are created automatically when computers join the domain.  
- **Domain Controllers container**: the default location for domain controller accounts.  
- **Users container**: where new users and groups are created by default.

 
**Domain Controllers (DCs)** are servers that host copies of the Active Directory database and provide essential network services:  
- They maintain a replica of the AD database and GPOs.  
- They provide **DNS services** to clients for name resolution and service discovery.  
- DCs offer **centralised authentication** via the **Kerberos** network security protocol.  

Functions of Domain Controllers:  
- Decide when users and computers can log onto the domain.  
- Control access to shared resources such as files and printers.

 
The speed and ease of network management through AD is highlighted:  
- When a new employee joins, a sysadmin can create a user account that is instantly recognised across all devices.  
- If an employee leaves or changes roles, disabling or deleting the account immediately adjusts access rights network-wide.


Most domain controllers are **read-write replicas**, meaning:  
- Each DC maintains a complete copy of the AD database.  
- All DCs can make changes, which are replicated to others.  
- Replication is generally fast and follows a **"last change wins"** rule.  

However, some changes require coordination to avoid conflicts. Such changes are handled by assigning **Flexible Single Master Operations (FSMO)** roles to specific DCs. FSMO roles ensure:  
- Certain critical changes are made safely by a single designated DC at a time.  
- Understanding FSMO roles and DC failure recovery is essential for domain controller management.


To leverage AD’s central authentication, computers must be **joined (or bound)** to the Active Directory domain. This process involves two key steps:  
1. AD creates and provisions a **computer account** for the joining machine.  
2. The computer recognises and authenticates against the Active Directory domain.  

After joining:  
- Computers authenticate with AD similarly to users logging onto the domain.  
- This integration ensures seamless network-wide authentication and policy application.

### Summary Table: Key Active Directory Components and Concepts

| Component                  | Description                                                                                          | Notes                                      |
|----------------------------|--------------------------------------------------------------------------------------------------|--------------------------------------------|
| **Active Directory (AD)**  | Native Windows directory service supporting LDAP for managing computers and users centrally.     | Introduced with Windows Server 2000.       |
| **LDAP Protocol**           | Lightweight Directory Access Protocol used by AD for interoperability with other OS.            | Supports Linux, OSX, etc.                   |
| **Group Policy Objects**    | Centralised configuration policies managed via AD.                                               | Enables consistent system configuration.   |
| **Active Directory Administrative Center (ADAC)** | Graphical tool for managing AD objects and configurations.                          | Used for everyday administrative tasks.    |
| **Objects**                 | Fundamental AD entities (users, computers, groups, containers).                                   | All AD elements are objects.                |
| **Containers**              | Simple containers holding objects; cannot contain other containers.                              | Default object storage locations.           |
| **Organisational Units (OUs)** | Containers that can nest other OUs or objects, enabling hierarchical organisation.           | More flexible than ordinary containers.     |
| **Domain**                  | Primary logical unit of AD, with short and DNS names.                                            | Domain objects have DNS names in the domain zone. |
| **Forest**                  | Collection of one or more domains sharing resources and accounts.                                | Domains in a forest trust each other.       |
| **Domain Controllers (DCs)**| Servers hosting AD database copies, providing authentication (Kerberos), DNS, and GPO services.  | Usually read-write replicas; critical for network security and management. |
| **Kerberos**                | Network security protocol used by DCs for authentication.                                       | Central to AD authentication.                |
| **FSMO Roles**              | Flexible Single Master Operations roles that assign exclusive rights to certain DCs for specific critical database changes. | Important for domain controller management and failure recovery. |
| **Computer Account**        | Represents a computer in AD, created when the machine joins the domain.                          | Necessary for network authentication and policies. |

---

### Key Insights  
- **Active Directory is foundational** to Windows network management, providing authentication, directory services, and policy enforcement.  
- The **hierarchical object model** (domains, OUs, containers) enables scalable and organised management of network resources.  
- **Domain controllers** are pivotal, hosting replicated databases and enforcing security through Kerberos authentication.  
- **Replication and FSMO roles** maintain data consistency and avoid conflicts in the AD database.  
- Joining devices to the domain integrates them fully into the organisation’s security and management framework.
