## Notes

 
This segment introduces **dnsmasq**, a lightweight, versatile program that provides multiple networking services including **DNS, DHCP, TFTP, and PXE** in a single package. It contrasts large enterprise environments, which often use separate programs for each service, with smaller setups that may benefit from a **centralised solution like dnsmasq**. The speaker prepares to demonstrate the installation and basic configuration of dnsmasq to enable hands-on understanding of these networking services.

 
Dnsmasq is installed using the command:  

```sudo apt install dnsmasq```  

Once installed, dnsmasq is **immediately enabled with basic functionality**, primarily acting as a **DNS caching server**. This means it stores the results of DNS queries locally, reducing the need to repeatedly query external DNS servers. The speaker introduces the **dig** command as a tool to test DNS queries, demonstrating how to query the local dnsmasq server for a domain address.

 
The command to query dnsmasq running on localhost is:  

```dig www.example.com @localhost```  

The `@localhost` specifies the DNS server to query (dnsmasq on the local machine). The response shows the IP address of the requested domain. To verify that dnsmasq is indeed handling the request, the speaker suggests running dnsmasq in **debug mode**, which provides detailed logs of DNS query handling.

 
To run dnsmasq in debug mode, the existing service must be stopped and restarted manually with debug flags:  
```sudo service dnsmasq stop```  
```sudo dnsmasq -d -q```  
- `-d` runs dnsmasq in debug mode (foreground execution with verbose output).  
- `-q` enables query logging.  
This setup is for **educational purposes** only, illustrating the internal workings of dnsmasq. On startup, dnsmasq outputs the **compilation options** and **configuration files** it is using.

 
With dnsmasq running in debug mode, the speaker repeats the DNS query using the same dig command:  

```dig www.example.com @localhost```  

The debug console shows that dnsmasq receives the query, forwards it to its configured upstream DNS server, and then responds to the client. When the same query is repeated, dnsmasq replies from its **local cache** instead of forwarding the request again, demonstrating the **caching mechanism** effectively reducing external DNS traffic.


Currently, dnsmasq functions as a **simple caching DNS server**. However, it can be configured to provide **authoritative DNS answers** for specific hostnames by supplying a list of hostname-to-IP mappings. This mirrors the functionality of an **ETC hosts file** but served via DNS. Authoritative DNS servers hold the definitive mappings, unlike caching servers which forward queries to authoritative sources.

  
The speaker shows the format of a custom hosts list file named `myhosts` using the command:  

```cat myhosts```  

The file contains simple IP-to-hostname mappings, for example:  
| IP Address     | Hostname        |  
|----------------|-----------------|  
| 192.168.1.10   | oxygen.local    |  
| *Not specified*| *Other entries* |  

Dnsmasq is instructed to use this hosts file with the `-H` flag:  

```sudo dnsmasq -d -q -H myhosts```  

This makes dnsmasq **authoritative for the hosts listed** in `myhosts`.


After loading the custom hosts file, a DNS query is made for one of the hostnames listed:  

```dig oxygen.local @localhost```  

Dnsmasq replies authoritatively without forwarding, confirming that it serves the hostname from its local data. The debug output confirms the source file used for the response.

 
For a hostname not in the local hosts file, such as:  

```dig hydrogen.local @localhost```  

Dnsmasq attempts to forward the query to the configured upstream DNS servers since it has no authoritative 
information. The response is an **NXDOMAIN** (non-existent domain) reply, indicating that the domain does not exist in the upstream DNS either. The debug output shows dnsmasq forwarding the query and receiving this negative response.

**Summary of dnsmasq functionality demonstrated:**  
- Provides **DNS caching** to improve query performance.  
- Can be run in **debug mode** for detailed logging of DNS query handling.  
- Supports **authoritative DNS service** for predefined host-to-IP mappings.  
- Forwards unknown queries to upstream DNS servers and handles NXDOMAIN responses.  
- Offers an easy, lightweight DNS solution suitable for small networks or testing environments.

**Key commands used:**  
| Command                                   | Purpose                                  |  
|-------------------------------------------|------------------------------------------|  
| `sudo apt install dnsmasq`                | Install dnsmasq                          |  
| `dig www.example.com @localhost`          | Query dnsmasq DNS server locally        |  
| `sudo service dnsmasq stop`                | Stop dnsmasq service                     |  
| `sudo dnsmasq -d -q`                      | Run dnsmasq in debug and query log mode |  
| `sudo dnsmasq -d -q -H myhosts`           | Run dnsmasq in debug mode with host file|

**Definitions:**  
| Term                 | Definition                                           |  
|----------------------|-----------------------------------------------------|  
| DNS caching          | Storing DNS query results locally to reduce repeated external queries |  
| Authoritative DNS    | DNS server with definitive answers for specific domain records |  
| NXDOMAIN             | DNS response indicating a domain does not exist     |  
| dig                  | Command-line tool for querying DNS servers           |  

This demonstration highlights the **simplicity and flexibility of dnsmasq**, making it a valuable tool for small network environments requiring DNS caching, DHCP, and PXE services in a unified package.
