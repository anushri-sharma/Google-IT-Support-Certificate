
# [DNS for Internal Networks](https://www.youtube.com/watch?v=zpzFs6QRtuk&t=3s)

🔥 **Hosts file = static, local IP-to-name mappings**  
  - **/etc/hosts** (Linux example) stores manual mappings such as **127.0.0.1 → localhost**.  
  - **localhost** commonly used to access a local web server.  
  - Editing the hosts file to map a public name (e.g., **www.google.com**) to **127.0.0.1** causes that name to resolve to the local machine, not the external site (demonstration: mapped name opened a local page).

🔥 **DNS resolution order: hosts file checked before DNS servers**  
  - A DNS lookup first consults the **hosts file**, then consults configured **DNS servers**.  
  - Any entry in the hosts file **overrides** DNS responses.

🔥 **Manual hosts-file edits do not scale**  
  - To map a machine like **catlady.examplecompany.com → 192.168.1.5** via hosts files, you must update the hosts file **on every computer** in the fleet.  
  - This approach is **tedious and not scalable** for larger networks.

🔥 **Local DNS server provides centralized name management**  
  - Run an internal **DNS server** that contains the organization's host-to-IP mappings as a **central storage location**.  
  - Update each computer’s network settings to use the **internal DNS server instead of the ISP’s DNS** to make the central records authoritative across the network.

🔥 **Directory service integration automates DNS population**  
  - Integrate DNS with a directory service (examples: **Active Directory**, **LDAP**) which centrally handles **user and machine information**.  
  - Once integrated, machine-to-IP mappings are **automatically populated** into DNS—**no manual entry** per machine required.

🔥 **Implications for internal network design**  
  - Use local DNS for **centralized, maintainable** host-name management.  
  - Integrate DNS with directory services to **automate** record creation and reduce administrative overhead.
