🔥 **Hosts file = static, local IP-to-name mappings**  
  - **/etc/hosts** (Linux example) stores manual mappings such as **127.0.0.1 → localhost**.  
  - **localhost** commonly used to access a local web server.  
  - Editing the hosts file to map a public name (e.g., **www.google.com**) to **127.0.0.1** causes that name to resolve to the local machine, not the external site (demonstration: mapped name opened a local page).

🔥 **DNS resolution order: hosts file checked before DNS servers**  
  - A DNS lookup first consults the **hosts file**, then consults configured **DNS servers**.  
  - Any entry in the hosts file **overrides** DNS responses.

🔥 **Manual hosts-file edits do not scale**  
  - To map a machine like **catlady.examplecompany.com → 192.168.1.5** via hosts files, you must update the hosts file **on every computer** in the fleet.  
  - This approach is **tedious and not scalable** for larger networks.

🔥 **Local DNS server provides centralized name management**  
  - Run an internal **DNS server** that contains the organization's host-to-IP mappings as a **central storage location**.  
  - Update each computer’s network settings to use the **internal DNS server instead of the ISP’s DNS** to make the central records authoritative across the network.

🔥 **Directory service integration automates DNS population**  
  - Integrate DNS with a directory service (examples: **Active Directory**, **LDAP**) which centrally handles **user and machine information**.  
  - Once integrated, machine-to-IP mappings are **automatically populated** into DNS—**no manual entry** per machine required.

🔥 **Implications for internal network design**  
  - Use local DNS for **centralized, maintainable** host-name management.  
  - Integrate DNS with directory services to **automate** record creation and reduce administrative overhead.

# Notes

**Overview of Managing Services in Linux Systems**  
- The video introduces the fundamental tasks a system administrator must perform regarding service management: **checking the status**, **starting**, **stopping**, and **restarting** services.  
- While specifics may vary by operating system, **the core concepts remain consistent across environments**.  
- The example used is the **Network Time Protocol (NTP)** daemon, which synchronizes the system clock on Ubuntu installations.  
- NTP runs as a background process (daemon) that ensures the machine’s clock remains accurate without manual intervention.

 
**Checking the Status of the NTP Service**  
- The command used is:  
  $$ \texttt{service ntp status} $$  
- This command confirms whether the NTP daemon is active and running on the machine.  
- The NTP service adjusts the clock continuously in **very small increments to avoid disruption of dependent services**.  
- Specifically, it adjusts time by **adding or subtracting 0.5 milliseconds per second** to keep the system clock in sync.  
- This gradual adjustment prevents sudden jumps in time which could negatively affect time-sensitive applications.

 
**Behaviour of the NTP Daemon with Clock Drift**  
- Under **normal conditions**, clock drift is minimal and corrected smoothly by the daemon.  
- If the clock difference exceeds **128 milliseconds**, the daemon assumes a larger external cause and **does not adjust the clock automatically**.  
- This threshold protects the system from abrupt or erroneous time corrections.  
- To demonstrate this, the video shows manually changing the system date to a past value using:  
  $$ \texttt{sudo date 2017-01-01 00:00:00} $$  
- After this manual modification, the date remains unchanged for several seconds because the offset exceeds the 128 ms threshold.

 
**Manual Date Change and NTP Non-Adjustment**  
- The system time is set to **1st January 2017, 00:00:00** manually.  
- Subsequent checks confirm the time remains at this past date rather than correcting itself immediately.  
- This illustrates the **NTP daemon’s safety mechanism to avoid sudden large corrections** during normal operation.

  
**Forcing the NTP Service to Correct Time**  
- NTP includes an option to **make large adjustments on service startup**, anticipating that no other services relying on time are running yet.  
- To apply this, the NTP service must be restarted, which triggers a drastic clock correction back to the present.  
- Commands used are:  
  - Stop the service:  
    $$ \texttt{sudo service ntp stop} $$  
  - Check current date (still incorrect):  
    $$ \texttt{date} $$  
  - Start the service:  
    $$ \texttt{sudo service ntp start} $$  
  - Check date again (now corrected):  
    $$ \texttt{date} $$  
- The video emphasises that **only users with administrative privileges (using sudo) can stop or start services**, while any user can check service status.

 
**Using Restart Command as a Shortcut**  
- Instead of stopping and starting separately, the **restart** command combines both actions:  
  $$ \texttt{sudo service ntp restart} $$  
- Demonstration steps:  
  1. Reset date back to 1st January 2017, 00:00:00:  
     $$ \texttt{sudo date 2017-01-01 00:00:00} $$  
  2. Confirm date is set:  
     $$ \texttt{date} $$  
  3. Restart NTP service:  
     $$ \texttt{sudo service ntp restart} $$  
  4. Confirm date is corrected:  
     $$ \texttt{date} $$  
- This method is more efficient and commonly used to refresh service states.


**Summary and Broader Application**  
- The video concludes by summarising how to manage services in Linux using the **service command with actions: status, start, stop, restart**.  
- Although NTP is a simple example, **these commands are applicable to managing complex services as well**.  
- Mastery of these commands is fundamental for system administrators to maintain service health and system stability.

### Key Commands and their Purposes

| Command                          | Purpose                                     | Notes                         |
|---------------------------------|---------------------------------------------|-------------------------------|
| $\texttt{service ntp status}$   | Check if the NTP service is running         | Can be run by any user         |
| $\texttt{sudo service ntp stop}$| Stop the NTP service                         | Requires administrative rights |
| $\texttt{sudo service ntp start}$| Start the NTP service                        | Requires administrative rights |
| $\texttt{sudo service ntp restart}$| Restart the NTP service (stop + start)    | Efficient for resetting service|

### Important Technical Insights

- **NTP adjusts system time in increments of 0.5 ms per second** to avoid disrupting dependent services.  
- **A threshold of 128 ms prevents NTP from making large time corrections automatically** during normal operation.  
- **Restarting the NTP service forces a large correction** to synchronise the clock with the real time, useful if the clock is manually changed or highly inaccurate.  
- **Use of `sudo` ensures only privileged users manage service states**, maintaining system security.
