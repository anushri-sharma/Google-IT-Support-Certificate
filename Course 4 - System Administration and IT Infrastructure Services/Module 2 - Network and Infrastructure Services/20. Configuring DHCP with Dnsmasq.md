
# [Configuring DHCP with Dnsmasq](https://www.youtube.com/watch?v=liaWHfArqps&t=2s)


## Notes

This segment introduces **dnsmasq** and its capability beyond DNS query serving, specifically highlighting its use as a **DHCP server**. A DHCP server typically runs on a machine with a **static IP address** configured on the network interface responsible for handling DHCP queries. This interface connects to the physical network where multiple client machines can obtain IP configuration dynamically.

- **Key terms:**  
  - **dnsmasq:** Lightweight DNS and DHCP server software  
  - **DHCP server:** Assigns IP addresses and network configurations to clients dynamically  
  - **Static IP:** Fixed IP address assigned to the server interface  
  - **Network interface:** The hardware or virtual interface connected to the network  

 
The tutorial simulates a DHCP server and client on a single machine. The server uses an interface called **eth_srv** configured with the IP address **192.168.1.1/24**. This subnet encompasses addresses from **192.168.1.0 to 192.168.1.255**. Although this interface also has an IPv6 address, only IPv4 is considered in this example.  

The client simulation uses the **eth_cli** interface, which initially lacks an IPv4 address. The goal is to assign an address to this interface via DHCP from the dnsmasq server running on eth_srv.

- **Network setup:**  
  | Interface | Role           | IP Address         | Notes             |
  |-----------|----------------|--------------------|-------------------|
  | eth_srv   | DHCP Server    | 192.168.1.1/24     | Server interface   |
  | eth_cli   | DHCP Client    | *None initially*   | Client interface   |

 
The next step involves configuring dnsmasq to serve DHCP requests. The configuration file (named **dhcp.config**) contains key directives:

- **interface=eth_srv:** Instructs dnsmasq to listen for DHCP queries only on the eth_srv interface.  
- **bind-interfaces:** Prevents dnsmasq from listening on other interfaces, allowing multiple dnsmasq instances to coexist on different interfaces without conflict.  
- **domain=network.domain:** Sets the network domain name provided to DHCP clients for hostname resolution.  

Two main DHCP options are provided to clients:  

- **default gateway:** The IP address clients should use as their router/gateway.  
- **DNS servers:** The addresses of DNS servers clients should use for resolving hostnames.

- **DHCP range:** Defines the pool of IP addresses available for assignment, avoiding the server’s own fixed IP.  

- **Lease time:** Specifies how long a client can retain an assigned IP address before it must renew. In this case, the lease time is set to **12 hours**.

| Configuration Aspect     | Description                                   | Example/Value          |
|-------------------------|-----------------------------------------------|-----------------------|
| interface               | Interface to listen on for DHCP queries       | eth_srv               |
| bind-interfaces         | Restricts listening to specified interface    | Enabled               |
| domain                  | Network domain name                            | network.domain        |
| DHCP options            | Default gateway, DNS servers                   | Configured accordingly|
| DHCP IP range           | Pool of assignable IP addresses                | Defined excluding server IP |
| Lease time              | Duration IP assigned before renewal            | 12 hours              |

 
The configuration avoids reserving static IPs for specific machines but notes this is possible. The DHCP range length is adjustable but must exclude the server’s IP. The lease time controls how long an IP remains allocated before being reclaimed if not renewed.


Dnsmasq is launched using the command:  
$$\texttt{sudo dnsmasq -d -q -c dhcp.config}$$  
- **-d:** Runs dnsmasq in the foreground (no daemonisation)  
- **-q:** Query logging enabled  
- **-c dhcp.config:** Specifies the configuration file to use  

Dnsmasq confirms it is listening for DHCP requests on **eth_srv** with the given configuration.

Simultaneously, a DHCP client is started on the **eth_cli** interface using:  
$$\texttt{sudo dhclient -i eth_cli -v}$$  
- **-i eth_cli:** Specifies the client interface  
- **-v:** Enables verbose output to observe DHCP packet exchanges  

This client-server interaction demonstrates the DHCP lease process, with the client successfully obtaining the IP address **192.168.1.80**.

 
The DHCP client output details the packet exchange sequence (DISCOVER, OFFER, REQUEST, ACK) and the lease renewal expectations. This confirms the standard DHCP protocol operation.

 
Verification of the client interface shows that **eth_cli now has the IP address 192.168.1.80** assigned by the DHCP server. Meanwhile, dnsmasq logs reveal it has learned the hostname corresponding to that IP address, leveraging its DNS capabilities.

- **Additional functionality:** dnsmasq acts as a combined DHCP and DNS server, providing authoritative DNS answers for local hostnames linked to leased IPs.


A DNS query is performed to validate dnsmasq’s DNS service:  
$$\texttt{dig @localhost instance-1}$$  
This confirms that dnsmasq responds authoritatively for local DNS queries, integrating DHCP and DNS services seamlessly.

- **Summary:** dnsmasq can function simultaneously as a DHCP and DNS server, simplifying network configuration and management.

 
The module concludes by emphasising practical considerations and broader learning points:  

- The presented setup is a **simulation on a single machine**; in real-world networks, DHCP and DNS servers typically run on **separate physical or virtual machines**.  
- Testing such configurations should be done on **isolated test networks** rather than production environments to avoid disruptions.  
- The module covered essential IT infrastructure services, including:  
  - Physical infrastructure services such as **remote access** and **virtualisation** to improve organisational efficiency.  
  - Core network services like **DNS** and **DHCP** and their roles in network management.  
  - The overall architecture and reasons for deploying DNS and DHCP servers within an organisation.

**Key takeaways:**  
- **dnsmasq** is a versatile tool that can provide both DHCP and DNS services.  
- Proper network interface configuration and DHCP options are critical for effective address management.  
- Lease times regulate address assignment and recycling.  
- Integrating DHCP with DNS simplifies local hostname resolution.  
- Real-world deployment requires separation of services and cautious testing.
