
# [Linux: Managing Processes](https://www.youtube.com/watch?v=4zB4jqcfVSU&t=7s)

Let's talk about how to use signals to manage processes in Linux.

First up, terminating processes.

**We can terminate a process using the kill command.**

It might sound a bit morbid, but that's just how it is in the dog eat dog world of terminating processes.

The kill command without any flags sends a termination signal or SIGTERM.

This will kill the process but it'll give it some time to clean up the resources it was using.

If you don't give the process a chance to clean up some of the files it was working with, it could cause file corruption.

I'm going to keep a process window open so you can see how our processes get affected as we run these commands.

So to terminate a process, we'll use the kill command along with the PID of the process we want to terminate.

Let's just go ahead and kill this Firefox process.

And if we check the process window, we can see that the process is no longer running.

The other signal that you might see pop up every now and then is the SIGKILL signal.

This will kill your process with a lot of metaphorical fire.

Using a SIG term is like telling your process, hey, there process, I don't really need you to complete right now, so could you just stop what you're doing?

And using SIGKILL is basically telling your process, okay, it's time to die.

The signal does its very best to make sure your process absolutely gets terminated and will kill it without giving it time to clean up.

To send a SIGKILL signal, you can add a flag to the kill command, -KILL for SIGKILL.

So let's open up Firefox one more time.

🔥 Terminate processes with SIGTERM  
  - **kill ** (no flags) sends **SIGTERM**, allowing the process time to clean up resources before exiting.  
  - Allowing cleanup reduces risk of **file corruption** or other resource inconsistencies.  
  - Example: killing a Firefox PID with **kill 12345** removes the process from the process list.

💥 Forcefully kill with SIGKILL  
  - **kill -KILL ** or **kill -9 ** sends **SIGKILL**, which terminates a process immediately without allowing cleanup.  
  - Example: **kill -KILL 10392** forcibly killed Firefox in the demo.  
  - Use **SIGKILL** only as a **last resort** because it can cause data loss or corrupted files.

⏸ Suspend processes with SIGTSTP  
  - **kill -TSTP ** sends **SIGTSTP** (terminal stop) and places a process into a **suspended** state.  
  - Keyboard shortcut **Ctrl‑Z** also sends **SIGTSTP** from the terminal.  
  - Use **ps -x** to view process statuses and verify a process is suspended.

Let's say you had a process running that you didn't want to terminate, but maybe you just want to put it on pause.

You can do this by sending the SIGSTP signal for terminal stop.

Which will **put your process in a suspended state.**

To send this, you can use the kill command with the flag -TSTP.

I'm going to **run ps -x** so you can see the status of the processes.

▶️ Resume processes with SIGCONT  
  - **kill -CONT ** sends **SIGCONT** to resume a suspended process.  
  - After **SIGCONT**, the process status will change (e.g., from **T** for stopped to **S** for resumed/sleeping as shown in the demo).

You can also send the SIGTSTP signal using the keyboard combination control -Z.

To resume the execution of the process, you can use the SIGCONT for continued signal.

🧭 Common signals and practical notes  
  - Key signals: **SIGTERM**, **SIGKILL**, **SIGTSTP**, **SIGCONT**.  
  - Default **kill** = **SIGTERM**; **SIGKILL** = immediate termination (often **-9**).  
  - Prefer graceful termination (**SIGTERM**) to avoid damaging files; use **SIGKILL** only when the process won’t respond.

🔧 Using signals to manage resources  
  - Suspend heavy or nonessential processes (**SIGTSTP**) to free CPU or I/O for higher-priority work.  
  - Resume or terminate processes as needed to optimize hardware utilization and maintain system stability.
