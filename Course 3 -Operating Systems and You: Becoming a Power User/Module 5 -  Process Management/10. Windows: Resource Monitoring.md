
# [Windows: Resource Monitoring](https://www.youtube.com/watch?v=eWZoptPvVCI&t=5s)

🔥 Why monitor processes  
  - **IT support roles** rely on process monitoring when processes become unruly; these skills are used frequently for diagnosing system issues.  
  - System tools surface which processes may be problematic by showing resource usage.

🪟 Windows Resource Monitor (GUI)  
  - Launch from the **Start menu**; interface shows **five tabs**: **Overview** plus one tab dedicated to each major resource (CPU, Memory, Disk, Network).  
  - Displays **process information** alongside the resources each process consumes, enabling correlation between a process and its resource use.  
  - Use Microsoft documentation for full column descriptions (e.g., memory columns).

🧭 Process Explorer — per-process performance graphs  
  - Access by selecting a process, **Right-click → Properties → Performance**.  
  - Visualizes **CPU**, **memory (Private Bytes)**, and **disk activity (I/O)** for the selected process.  
  - Provides a **less detailed** but quick per-process view compared with Resource Monitor.

🖥️ Command-line: PowerShell Get-Process (overview)  
  - Running **Get-Process** with no parameters returns a row per running process and many columns of metrics.  
  - Example column **NPM(K)** (commonly shown as **NPM(K)**) indicates **nonpaged memory** used by the process; **K = kilobytes**.  
  - The default output can be verbose; useful to filter or select specific fields for focus.  
  - Official docs list full meanings of all columns and units.

🔎 Example: show top three CPU consumers (command + breakdown)  
  - Command (canonical form): **Get-Process | Sort-Object CPU -Descending | Select-Object -First 3 -Property ID, ProcessName, CPU**  
  - Step-by-step:  
    - – **Get-Process**: retrieve all process objects from the OS.  
    - – Pipe to **Sort-Object CPU -Descending**: order processes by the **CPU** column, largest first.  
    - – Pipe to **Select-Object -First 3 -Property ID, ProcessName, CPU**: take the top **3** rows and show only **ID**, **ProcessName**, and **CPU** for concise output.  
  - Pipes connect commands so output flows from one cmdlet to the next (same piping concept as in many Unix/Linux shells).
