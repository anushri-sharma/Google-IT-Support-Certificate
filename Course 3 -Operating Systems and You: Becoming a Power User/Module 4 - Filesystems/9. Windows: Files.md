🔥 **OS manages file data, file metadata, and file systems**  
  - **File data** = the actual contents of a file (e.g., text in a document).  
  - **File metadata** = attributes like **owner**, **permissions**, **size**, **creation timestamp**, **location on disk**, **read-only/compressed flags**, and other properties.  
  - **NTFS** is the **native Windows file system**; the OS uses it to store both data and metadata.

💡 **Master File Table (MFT) is NTFS’s central index**  
  - **Every file on a volume has at least one MFT entry**, and the MFT itself is represented inside the MFT.  
  - There is usually a **one-to-one correspondence** between files and MFT records; if a file has many attributes it may use **multiple MFT records**.  
  - When a file is deleted, its MFT entry is **marked free for reuse** (the record can be overwritten later).

📌 **MFT entries store file attributes and identifiers**  
  - Attributes include **file name**, **timestamps**, **flags** (read-only, compressed), **location of the file’s data**, and more.  
  - A key identifier in an MFT entry is the **file record number** — the index of that file’s entry in the MFT.

📁 **Shortcuts (.lnk) are just files with a reference**  
  - A **shortcut** is a regular file with an MFT entry that contains a reference to a destination.  
  - Creating a shortcut: **right-click the target → Create shortcut**.  
  - Behavior example: opening a shortcut file with Notepad opens the shortcut file itself (binary/structured data), not the linked file’s contents.

🔗 **Symbolic links behave like substitutes at the file-system level**  
  - A **symbolic link** (symlink) is an MFT entry that points to **the name/path of another file entry**.  
  - The OS treats a **symlink like the original file in almost every meaningful way** (opening a symlink accesses the target’s contents).  
  - Example commands (Windows): **mklink file_1_symlink file_1** (creates a file symlink).  
  - Demonstration: a symlink to a text file opened in Notepad displays the target’s text (“Hello”), unlike a shortcut.

🧷 **Hard links point to the file record number, not the name**  
  - A **hard link** is an MFT entry that references the target’s **file record number** (the MFT index).  
  - Because it points to the record number, the **target file’s name can change and the hard link still resolves** to the same file data.  
  - Create a hard link with mklink: **mklink /H file_1_hardlink file_1**.  
  - Implication: hard links are effectively multiple directory entries for the same file content; deleting one name does not remove the data if other links remain.

When you create a hard link in NTFS, an entry is added to the MFT that points to the linked file record number, not the name of the file.

This means the file name of the target can change and the hard link will still point to it.

You can create hard links in a way that's similar to symbolic links, but with the /H option.

So make link /H file_1_hardlink file_1.

Since a hard link points out the file record number and not the file name, you can change the name of the original file and the link will still work.

🛠️ **Tooling and practical notes**  
  - Use **mklink** from an elevated Command Prompt to create symlinks/hard links (older Windows required admin; newer Windows can allow non-admin symlink creation with Developer Mode enabled).  
  - Choose between shortcut, symlink, and hard link depending on needs:  
    - **Shortcut** = simple reference file for user navigation (does not act as the target to programs).  
    - **Symlink** = name-based proxy that programs treat like the target.  
    - **Hard link** = direct additional directory entry bound to the target’s MFT record number (robust to target renames).

🧾 **Edge cases and storage behavior**  
  - A file with many attributes may consume **multiple MFT records**, complicating forensic or recovery scenarios.  
  - Deleted MFT entries being marked free means metadata can be **reused/overwritten**, affecting recovery chances.  
  - Choice of link type affects behavior under renames, deletions, and program access (name-based vs record-based resolution).
