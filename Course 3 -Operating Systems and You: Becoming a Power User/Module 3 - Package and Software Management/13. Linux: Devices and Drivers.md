🔌 **Device files: hardware exposed as files in /dev**
  
  – **Linux treats hardware devices as files**; when a device is connected a corresponding device file is created in **/dev**.  
  – **Not all entries in /dev are physical devices** (example: **/dev/null**).  
  – Implication: standard file operations and tools can be used to interact with devices.

- **In Linux, everything is considered a file, even hardware devices.**
- **When a device is connected** to your computer, **a device file is created in the /dev directory.**

🧾 **File-type indicators in ls -l show device types**
  
  – The first character in **ls -l** output denotes file type: **-** (regular file), **d** (directory), **c** (character device), **b** (block device).  
  – Use this to distinguish device files from regular files and directories.

⌨️ **Character devices: byte/character-oriented I/O**

  <img width="1274" height="373" alt="image" src="https://github.com/user-attachments/assets/af3459b1-f3c3-4e0c-bddd-81b6a3c964c1" />

  – **Character devices** transmit data **character by character** (examples: keyboard, mouse, **/dev/null**).  
  – Use-cases: streaming, byte-wise interaction; represented with a **c** in file-type field.

💽 **Block devices: block-oriented storage devices**
  
  – **Block devices** transfer data in fixed-size **blocks** (examples: USB drives, hard drives, CD-ROMs).  
  – Represented with a **b** in file-type field.  
  – A data block is a unit of storage used by block devices and filesystems.

🗂️ **Mass-storage device naming: /dev/sdX**

<img width="882" height="339" alt="image" src="https://github.com/user-attachments/assets/2d90fc2f-aeeb-48ba-8575-327cb192c6bc" />

  
  – Mass-storage devices appear as **/dev/sdX** (examples: **/dev/sda**, **/dev/sdb**, **/dev/sdc**).  
  – The letter suffix (**a**, **b**, **c**) reflects detection order; **a** = detected first.  
  – Implication: multiple drives and removable media map to sequential sd names.

🧠 **Where drivers live: kernel vs /dev**
  
  – **Device drivers are not stored in /dev**; **/dev** holds device files only.  
  – The **Linux kernel** handles hardware interaction; it is a largely monolithic component that contains much hardware support.  
  – Many devices work automatically because their drivers are **built into the kernel**.

🧩 **Kernel modules: extend kernel functionality without modifying it**
  
  – When hardware support is not built-in, support often comes as a **kernel module** (loadable module).  
  – Kernel modules let developers add functionality without altering the core kernel source.  
  – You can install kernel modules using standard Linux software installation methods (package managers, distro packages).  
  – Caution: **not all kernel modules are device drivers**; modules can provide other kernel-level features.

🔄 **Driver updates in Linux vs Windows**
  
  – Windows often uses a simple “Update driver” GUI; Linux management differs because drivers may be integrated in the kernel or provided as modules.  
  – Updating drivers in Linux can involve installing/updating modules or upgrading the kernel/distribution packages.  
  – Practical implication: driver updates may be both more flexible (modules) and more involved (kernel/module management) than the single-button Windows approach.

ℹ️ **Scope note**
  
  – There are many device types beyond the basics covered here; this is a concise overview focusing on the common character and block device concepts and driver handling.
