
# [Windows: Package Dependencies](https://www.youtube.com/watch?v=2PYfsqJkrLo&t=3s)


🔗 **Dependencies: programs rely on other software components to work**  
  – Example: a game may depend on a **physics engine** for calculations and a **rendering library** for graphics.  
  – Implication: all dependent components must be available to the program at runtime or the program will fail.

📦 **Library concept and Windows shared libraries (DLLs)**  
  – A **library** bundles reusable code so multiple programs can "tap into" the same functionality.  
  – In Windows, shared libraries are **dynamic link libraries (DLL)**.  
  – Benefit: the same DLL can be used by many programs, reducing overall memory usage because shared code need not be loaded separately for each process.

🖥️ **Installers and bundled dependencies (MSI and installation packages)**  
  – Windows applications commonly ship dependencies (DLLs, other resources) together in a single installation package and use an **MSI** file to instruct the Windows installer how to deploy them.  
  – Installer behavior: bundled dependencies are installed so the application has everything it needs without manual, per-dependency installation.

⚠️ **Historical problem: "DLL hell" and version conflicts**  
  – Scenario: an existing app (e.g., a video player) uses a particular graphics DLL; a newly installed game updates that DLL to a new version incompatible with the player, causing the player to break.  
  – Effect: previously working programs can fail when shared libraries are overwritten or replaced by incompatible versions.

🧩 **Solution on modern Windows: side-by-side assemblies (SxS)**  
  – Shared libraries and related resources are managed as **side-by-side assemblies (SxS)**.  
  – Primary storage: **C:\Windows\WinSxS**.  
  – Mechanism: applications specify required libraries and versions in a **manifest**; Windows loads the appropriate versions from WinSxS.  
  – Capability: SxS supports multiple versions of the same library concurrently, preventing installers from "pulling the rug out" from under existing programs.

🧰 **Package management via PowerShell cmdlets**  
  – A **cmdlet** is a PowerShell command using the verb-noun format (examples: **Get-Help**, **Select-String**).  
  – Windows provides package-management cmdlets such as **Find-Package** and **Install-Package** to locate and install software plus dependencies from configured package sources.  
  – Example usage:  
    – Use **Find-Package sysinternals -IncludeDependencies** to locate the Sysinternals package and its dependencies.  
    – If the default source doesn’t contain the package, add a repository with **Register-PackageSource**.

🔗 **Repository example: adding Chocolatey as a package source**  
  – Problem: the default **PowerShell Gallery** may not have certain packages (e.g., **Sysinternals**), causing Find-Package to return "no match found."  
  – Solution: add Chocolatey as a source:  
    – Command: **Register-PackageSource -Name chocolatey -ProviderName Chocolatey -Location https://chocolatey.org/api/v2**  
    – Verify sources with **Get-PackageSource**.  
    – After adding the source, **Find-Package sysinternals -IncludeDependencies** should locate the package.  
  – Final installation: use **Install-Package** (optionally with **-IncludeDependencies**) to install Sysinternals and its dependencies.

❗ **Practical caveats and implications for administrators/users**  
  – Always ensure relevant package sources are registered before searching or installing; missing sources cause command errors.  
  – Package managers simplify dependency resolution, but system-level mechanisms (SxS, WinSxS) and installer behavior still matter for compatibility.  
  – When troubleshooting broken applications after installs, check for library version mismatches, manifests, and installed package sources.
