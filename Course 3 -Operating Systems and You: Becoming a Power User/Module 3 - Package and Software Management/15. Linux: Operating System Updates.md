🔥 **Kernel vs regular packages**  
  – **Kernel** controls the core components of the operating system but is managed as **just another package** in Linux.  
  – Kernel updates include **security patches, new features, and bug fixes**; to receive those changes you must install and run a newer kernel version.

💡 **Check current kernel version**  
  – Use **uname -r** to show the kernel release (example given: **4.1**).  

🛠️ **Package update workflow (apt)**  
  – Refresh package sources with **sudo apt update**.  
  – **sudo apt upgrade** installs available package updates and may include some security updates, **but it does not always upgrade the kernel or other packages that require adding/removing packages**.  
  – Use **sudo apt full-upgrade** (equivalent to older **apt-get dist-upgrade**) to allow installation of new kernel packages and resolve dependency changes that **apt upgrade** will not perform.  
  – Expect prompts: **full-upgrade** can install or remove packages to satisfy upgrades; review prompts before confirming.

🔁 **Activate a newly installed kernel**  
  – After a kernel package is installed, a **reboot** is required for the system to run the new kernel.  
  – Verify the active kernel after reboot with **uname -r**.

⚠️ **Caveats and additional details**  
  – Kernel installation and security-update mechanisms have extra details not covered here (e.g., package naming, bootloader entries, multiple installed kernels).  
  – Some deployments use **live patching** (for example, Canonical Livepatch, kpatch) to apply certain kernel fixes without rebooting.  
  – Behavior and tooling can vary by distribution and kernel packaging policy; always follow distro-specific guidance and check upgrade prompts and changelogs.
