🔍 Understanding why IT should care  
  – **Installing software can change system state in ways you must be able to diagnose**, e.g., a package modifying a configuration file it shouldn’t and causing failures.  
  – Knowing underlying mechanisms lets you inspect, debug, and revert problematic changes.

🔧 Custom EXE installers (closed‑source)  
  – **Custom EXE installer** behavior depends entirely on how the developer implemented installation logic.  
  – Most Windows desktop software is **closed‑source**, so you cannot read installer code to see what it will do.  
  – To inspect actions of such installers, run monitoring tools rather than reading source.

🧭 Monitoring installer activity: Process Monitor  
  – **Microsoft Sysinternals Process Monitor** can show runtime installer activity.  
  – It reports file writes, registry changes, and process activity performed by the installer, enabling forensic inspection of what the installer actually did.

🗄️ MSI installers: structured, rule‑based packages  
  – **MSI file** is not a single simple file but a set of **relational database tables** plus embedded files, resources, shortcuts, libraries, and other objects the app needs.  
  – The **Windows Installer** interprets the **tables in the MSI database** to know where to place files and what configuration steps to perform.  
  – An EXE can simply be a wrapper around an MSI; the underlying MSI still follows the same structured format.

🔁 Uninstallability and rollback logic  
  – The Windows Installer **records all actions it takes** during installation.  
  – It **creates a separate set of instructions to undo** those actions, which is how uninstallation and rollback are implemented.

🛠️ Inspecting and editing MSIs: Orca.exe  
  – **Orca.exe** (part of the Windows SDK) lets you open, inspect, edit, or create MSI packages by viewing and modifying their tables.  
  – **You do not need to be a programmer** to use Orca; it’s useful for curiosity, troubleshooting, or custom packaging.  
  – Using Orca you can verify or change install instructions before deployment.

⚠️ Practical implications and recommended actions for IT  
  – For opaque EXE installers, use runtime monitoring (Process Monitor) to detect unexpected file/registry changes.  
  – Prefer MSI‑based packages for predictable, standard installation behavior and reliable uninstall/rollback.  
  – Use **Orca.exe** to audit or modify MSI tables when packaging for enterprise deployments or correcting misbehaving installers.  
  – Be aware that installers may alter critical configuration files; verify modifications in staging before production rollout.
