
# [The Windows Event Viewer](https://www.youtube.com/watch?v=PMBpf4nba48&t=6s)

🔥 **What Event Viewer is and when to use it**  
  - In Windows, the events logged by the operating system are stored in an application called The Event Viewer.
  – **Event Viewer** stores operating system and application events on Windows and is a primary troubleshooting starting point for issues like **application crashes**, **login/access problems**, or general system diagnostics.  
  – Practical uses include diagnosing a game crash, investigating failed driver startup, or auditing who accessed the computer.

🖱️ **How to open Event Viewer**  
  – Launch from the **Start menu** or run **eventvwr.msc** from the Run box.

🗂️ **Default view vs. left-hand navigation**  
  – The default view shows a **summary of potentially important recent events**, which may be noisy for focused troubleshooting.  
  – The left-hand pane contains three main groupings: **Custom Views**, **Windows Logs**, and **Applications and Services Logs**.

🔎 **Custom Views (filtering across logs)**  
  – **Custom Views** let you create a cross-log filter to extract just the events you care about instead of sifting through everything.  
  – Example step-by-step filter: click **Create custom view** in the Actions pane → open the **Filter** tab → check **Error** and **Critical** → set the time range dropdown to **Last hour** → choose **Windows Logs** → click **OK** → give the view a name.  
  – Result: a new view under **Custom Views** showing only events that match the filter.

📚 **Windows Logs (system-wide events)**  
  – **Windows Logs** contain events relevant to the whole OS.  
  – Examples and implications:  
    – **System log** is the first place to check for **driver failures during startup**.  
    – **Security log** is where to investigate **who has been accessing the computer** (audit/login/access events).

🧩 **Applications and Services Logs (per-app/component)**  
  – These logs track events from a **single application or OS component** rather than system-wide events.  
  – Example: check the **PowerShell** log under Applications and Services Logs when troubleshooting PowerShell-specific issues.

📄 **Structure of an event entry and what to inspect**  
  – Each line in a log is an **event** with columns such as **logging level**, **date/time**, and other identifiers.  
  – **Logging levels** range from **Information** (lowest) up to **Critical** (highest).  
  – Selecting an event displays **detailed information in the bottom pane** that helps with troubleshooting or preparing a bug report.

🛠️ **Practical advice and limitations**  
  – Event Viewer provides **detailed diagnostic data** useful for IT support and troubleshooting hardware or software problems.  
  – The volume of information can be large; rely on **custom views and filters** to isolate relevant events.  
  – Familiarity with the interface improves effectiveness; explore logs and filters to learn typical event patterns and relevant sources.
