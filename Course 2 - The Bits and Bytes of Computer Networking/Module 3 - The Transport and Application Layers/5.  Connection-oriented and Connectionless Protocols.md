
# [Connection-oriented and Connectionless Protocols](https://www.youtube.com/watch?v=JWEguvjfsjM&t=4s)

🔥 **Connection-oriented protocols (example: TCP)**  
  – **TCP** establishes a connection and uses a stream of **acknowledgements (ACKs)** so each endpoint knows which data arrived.  
  – ACKs let the sender detect missing data and decide when to **retransmit**; retransmission decisions are made by the transport layer, not lower layers.  
  – **Sequence numbers** label data so segments can be reassembled in order even if they arrive out of order.  
  – This approach protects against many real-world failures: bit errors, router drops from congestion, physical link cuts, and other delivery failures.  
  – Clarification: TCP actually acknowledges bytes/segments of data, not individual bits; sequence numbers and ACKs track what has been delivered.

🧩 **Checksums and responsibility boundaries**  
  – Lower layers (IP, Ethernet) use **checksums** to detect corrupted frames and **discard** bad data; they do not attempt retransmission.  
  – Because lower layers drop corrupted data, the transport layer (e.g., TCP) must detect missing data via ACKs/sequence numbers and handle retransmission.

⚖️ **Costs and overhead of connection-oriented operation**  
  – Costs include **connection setup**, a continuous stream of ACKs during the session, and **connection teardown**—all of which add extra network traffic.  
  – That overhead is justified when **reliability is required** (e.g., file transfer, web page loads) but is wasteful if perfect delivery is unnecessary.

⚡ **Connectionless protocols (example: UDP)**  
  – **UDP (User Datagram Protocol)** sends packets to a destination port with **no connection** and **no acknowledgements**.  
  – The sender transmits datagrams and has no built-in mechanism to know whether they arrived or were corrupted.  
  – This reduces overhead and can free bandwidth for payload data instead of control traffic.

🎯 **Use cases and implications (examples and trade-offs)**  
  – Streaming media (illustrative: treating each UDP datagram as a video frame) tolerates some loss: missing a few frames usually still yields an acceptable experience.  
  – Using UDP can enable **higher effective throughput** for lossy-tolerant applications because it avoids TCP’s connection and ACK overhead.  
  – Trade-off: reduced overhead comes at the cost of **no built-in reliability**, so applications must tolerate loss or implement their own recovery if needed.  
  – Note: real-world deployments may still use TCP/HTTP for services like YouTube; the streaming-UDP example is a conceptual illustration of when connectionless delivery can be preferable.

🔍 **Edge cases and additional considerations**  
  – Out-of-order arrival is harmless for TCP because **sequence numbers** restore order; for UDP, the application must handle ordering if needed.  
  – Applications that need both low overhead and some reliability can implement custom mechanisms over UDP (e.g., selective retransmit, FEC), but that shifts complexity to the application layer.
