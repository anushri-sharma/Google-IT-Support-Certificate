
# [Dissection of a TCP Segment](https://www.youtube.com/watch?v=YjXgVQK5b7A&t=4s)



In IT support, if network traffic isn't behaving as users expect it to, you might have to analyze it closely to troubleshoot.

Get ready to take a peek at all the inner workings.

Just like how an Ethernet frame encapsulates an IP datagram, an IP datagram encapsulates a TCP segment.

Remember that an Ethernet frame has a payload section, which is really just the entire contents of an IP datagram.

Remember also that an **IP datagram** has a payload section, and this is made up of what's known as a **TCP segment.**

A **TCP segment** is made up of a **TCP header and a data section**.

This data section, as you might guess, is just another payload area for where the application layer places its data.

A **TCP header itself is split into lots of fields containing lots of information.**

🔥 Encapsulation hierarchy: Ethernet → IP → TCP  
  – **Ethernet frame payload** contains an **IP datagram**; the **IP datagram payload** is a **TCP segment**.  
  – A **TCP segment** = **TCP header** + **data payload** (the application-layer data).

🍳 TCP header top-level fields: ports  
  – **Destination port** identifies the service on the remote host (example: **port 80** for HTTP).  
  – **Source port** is a high-number chosen from the **ephemeral ports** range to keep outgoing connections distinct so replies are delivered to the correct local application (e.g., web browser vs word processor).

💡 Sequence and acknowledgment control  
  – **Sequence number** is a **32-bit** value that identifies where this segment fits in the stream of TCP bytes; used because lower-layer size limits force splitting data across many segments (typical Ethernet frame ≈ **1518 bytes**).  
  – **Acknowledgment number** indicates the **next expected sequence number** (example: **seq=1**, **ack=2** reads as “this is segment 1; expect segment 2 next”).  
  – These fields implement ordered, reliable delivery across multiple segments.

🔧 Header layout details: length and flags  
  – **Data offset** is a **4-bit** field that states the **TCP header length** (measured in 32-bit words) so the receiver knows where the data payload begins (max header length = **60 bytes**).  
  – **Six TCP control flags** occupy six bits; common flags: **URG, ACK, PSH, RST, SYN, FIN**.  
    – URG marks urgent data (works with urgent pointer).  
    – ACK acknowledges received data.  
    – PSH requests immediate delivery to the application.  
    – RST resets a connection.  
    – SYN initiates sequence number synchronization (connection setup).  
    – FIN signals connection close.

In very simple language, a sequence number of one and an acknowledgment number of two could be read as this is segment 1, expect segment 2 next.

The data offset field comes next.

This field is a four-bit number that communicates how long the TCP header for this segment is.

This is so that the receiving network device understands where the actual data payload begins.

Then we have six bits that are reserved for the six TCP control flags.

The next field is a 16-bit number known as the TCP window.

A TCP window specifies the range of sequence numbers that might be sent before an acknowledgment is required.

TCP is a protocol that's super reliant on acknowledgments.

This is done in order to make sure that all expected data is actually being received and that the sending device doesn't waste time sending data that isn't being received.

📐 Flow control: TCP window  
  – **TCP window** is a **16-bit** field that specifies how many bytes (the range of sequence numbers) the sender may transmit before receiving an acknowledgment.  
  – Windowing enforces receiver-side flow control and prevents the sender from overwhelming the receiver; acknowledgments are central to TCP’s reliability and efficiency.

🛡 Integrity: checksum  
  – **Checksum** is a **16-bit** field computed over the segment (plus protocol pseudo-header at IP layer) and compared on receipt to detect corruption or loss in transit.

⚠️ Urgent pointer (rare in practice)  
  – **Urgent pointer** works with the **URG** flag to indicate a location of urgent data within the stream.  
  – This mechanism is historically available but seldom used in modern network stacks.

<img width="1279" height="462" alt="image" src="https://github.com/user-attachments/assets/aa35f298-d2aa-43e1-ab21-a589fd148910" />


🧩 Options field and padding  
  – **Options field** allows optional extensions (commonly used options include **MSS**, **Window Scale**, **Timestamps**, **SACK** for performance and flow-control enhancements).  
  – **Padding** (zeros) aligns the header so the data payload begins on a 32-bit boundary and the header meets the length indicated by the **data offset**.
