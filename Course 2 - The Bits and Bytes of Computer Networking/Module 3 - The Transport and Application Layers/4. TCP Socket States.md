
# [TCP Socket States](https://www.youtube.com/watch?v=O4I4Qjguiqc&t=6s)

### A socket is the instantiation of an endpoint in a potential TCP connection.

<img width="1265" height="365" alt="image" src="https://github.com/user-attachments/assets/b7368c1c-98c7-4a68-add2-48c05a42def2" />

### An instantiation is the actual implementation of something defined elsewhere.

<img width="1263" height="323" alt="image" src="https://github.com/user-attachments/assets/7d1758fe-94d4-4a50-9609-3e3438d19085" />

TCP sockets require actual programs to instantiate them.

You can contrast this with a port, which is more of a virtual descriptive thing.

In other words, you can send traffic to any port you want, but you're only going to get a response if a program has opened a socket on that port.

TCP sockets can exist in lots of states, and being able to understand what those mean will help you troubleshoot network connectivity issues as an IT support specialist.


### Listen.

Listen means that a TCP socket is ready and listening for incoming connections.

<img width="1275" height="353" alt="image" src="https://github.com/user-attachments/assets/4b88b88d-ae75-452a-bf7c-c487055fe9fe" />

You'd see this on the **server side only**.

### SYN_SENT.

This means that a synchronization request has been sent, but the connection hasn't been established yet.

<img width="1280" height="344" alt="image" src="https://github.com/user-attachments/assets/3d758815-8f10-4d73-b464-b5f8f8f5f749" />

You'd see this on the **client side only**.

### SYN_Received.

This means that a socket previously in a listen state has received a synchronization request and sent a SYN/ACK back.
<img width="1277" height="427" alt="image" src="https://github.com/user-attachments/assets/c496ecaf-e211-4df8-91bd-487e0e847d11" />

But it **hasn't received the final ACK from the client yet**.

You'd see this on the **server side only**.

### Established.

This means that the TCP connection is in working order and both sides are free to send each other data.

<img width="1274" height="353" alt="image" src="https://github.com/user-attachments/assets/3fae589e-aca0-4e32-9eeb-69a45fb485c3" />

You'd see this state on both the client and server sides of a connection.

This will be true of all the following socket states, too, so keep that in mind, 

### FIN_WAIT.

This means that a fin has been sent, but the corresponding **ACK from the other end hasn't been received yet**.

<img width="1272" height="353" alt="image" src="https://github.com/user-attachments/assets/7b675abb-9364-4f6d-b93f-c59c4334a261" />

### Close_Wait.

This means that the connection has been closed at the TCP layer, but that the application that opened the socket hasn't released its hold on the socket yet.

<img width="1273" height="454" alt="image" src="https://github.com/user-attachments/assets/c4429026-5b9f-4036-af1e-eeab869846a2" />


### Closed.

This means that the connection has been fully terminated and that no further communication is possible.

There are other **TCP socket states that exist.**

<img width="1271" height="381" alt="image" src="https://github.com/user-attachments/assets/94adcca4-df56-49d7-95f0-7b5ff796dc14" />

Additionally, socket states and their names can vary from operating system to operating system.

That's because they exist outside of the scope of the definition of TCP itself.

TCP, as a protocol is universal in how it's used since every device speaking the

**TCP protocol** has to do this in the exact same way for communications to be successful.

Choosing how to describe the state of a socket at the operating system level isn't quite as universal.

When troubleshooting issues at the TCP layer, make sure you check out the exact socket state definitions for the systems you're working with.
