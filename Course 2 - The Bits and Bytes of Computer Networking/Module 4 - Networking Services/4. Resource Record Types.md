
# [Resource Record Types](https://www.youtube.com/watch?v=LE0QAHhrCKM&t=10s)

## 🔥 DNS importance & resource record overview

- **DNS** enables name-to-resource mappings and is essential for troubleshooting network issues.
- There are **dozens of resource record (RR) types**, many specialized; this summary covers the most common, broadly used types.

## 🍳 A record (Address record) — IPv4 mapping

- **A record** maps a domain name to an **IPv4 address**.

<img width="1278" height="350" alt="image" src="https://github.com/user-attachments/assets/d24dbbd2-d0cc-4c8d-8454-807f303bb4d2" />

- A single name can have **multiple A records**, enabling simple load distribution.
- Example: **www.microsoft.com** configured with A records **10.1.1.1, 10.1.1.2, 10.1.1.3, 10.1.1.4.**


## 💡 DNS round-robin behavior (with A records)

<img width="1280" height="386" alt="image" src="https://github.com/user-attachments/assets/091037ba-e116-4437-97e8-7fd7a64733ba" />


- A resolver receives **all configured IPs** in the order they were stored (e.g., 10.1.1.1, 10.1.1.2, 10.1.1.3, 10.1.1.4).
- The client typically tries the **first** entry but knows fallback addresses.
- Subsequent responses can **rotate the ordering** (10.1.1.2 first, then 10.1.1.3, …), cycling through entries to **balance traffic** across IPs.

## 🧭 AAAA (Quad-A) record — IPv6 mapping

- **AAAA (Quad A**) is analogous to an A record but returns an **IPv6 address** instead of IPv4.

## 🔁 CNAME (Canonical Name) — aliasing / redirection

<img width="1279" height="391" alt="image" src="https://github.com/user-attachments/assets/2a0b6ff6-98c0-41b5-af08-48585d088c54" />


- **CNAME points** one domain name to another domain name (the canonical name), causing the resolver to perform a second lookup for that canonical name.
- Example: point **microsoft.com → www.microsoft.com** so both names resolve to the same final IP(s).
- Benefit: **single source of truth** for the canonical name’s A/AAAA records, simplifying updates when IPs change.

<img width="1276" height="313" alt="image" src="https://github.com/user-attachments/assets/89ffdb63-a8f4-4bc8-aac2-47ff994bde6b" />


## ✉️ MX (Mail Exchange) — email delivery routing

- **MX** records specify the **mail server(s)** responsible for accepting email for a domain.
- Common when web and mail services run on **different hosts/IPs**, so web traffic and mail delivery can be routed separately.

<img width="1278" height="384" alt="image" src="https://github.com/user-attachments/assets/bbe67eeb-613e-45fe-9b36-6283b95750c4" />

## 🛠️ SRV (Service) — service-specific location records

- **SRV** records define the **hostname, port, priority, and weight** for a specific service (not limited to mail).
- Used for services such as **caldav, VoIP,** and other discoverable services where clients need service-specific connection details.

<img width="1274" height="349" alt="image" src="https://github.com/user-attachments/assets/0caeb73c-aae3-4f69-9bcb-6ffe759c064b" />


## 📜 TXT (Text) — free-form text, now machine-parsed

- **TXT** was intended for human-readable notes about a domain but is now widely used to convey machine-readable configuration data.
- Common uses include delegating configuration to third parties (e.g., **email-as-a-service providers**) and storing service-related metadata such as SPF/DKIM/DMARC data.

<img width="1274" height="427" alt="image" src="https://github.com/user-attachments/assets/3df713fe-a723-4c0d-a976-4b9ae041d08e" />


## 🗂️ Other common zone records: SOA and DNS

- **SOA (Start of Authority)** contains authoritative zone metadata (serial, refresh/ retry/TTL values, authoritative contact).
- **DNS (Domain Name Server)** records identify the authoritative name servers for a zone.


## ⚠️ Operational implications & practices

- Using **CNAMEs** reduces duplication of A/AAAA records across many names and centralizes updates.
- **Round-robin** offers simple load distribution but does not perform health checks; failed hosts can still be handed out unless additional mechanisms are used.
- TXT records are flexible and often used for third-party integrations; their free-form nature enables creative but inconsistent usages across services.
