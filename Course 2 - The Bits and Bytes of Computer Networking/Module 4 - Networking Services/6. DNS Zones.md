🔥 **What a DNS zone is**
  
  - **DNS zones are hierarchical administrative units**; an authoritative name server is responsible for a specific zone.  
  - The **root** and **TLD** name servers are also authoritative servers whose zones are special cases (root zone, each TLD zone).  
  - **Zones do not overlap**: authority stops at the delegated child zone (e.g., the .com TLD server does not administratively control google.com once google.com is delegated).

💡 **Why zones exist (purpose and benefits)**
  
  - Zones let administrators **split large domains** to simplify management and delegation.  
  - Example: a company owning **largecompany.com** with offices in Los Angeles, Paris, Shanghai and ~200 desktops per office would face **600 A records** if managed as one zone.  
  - Splitting into subdomain zones (e.g., **la.largecompany.com**, **pa.largecompany.com**, **sh.largecompany.com**) distributes the record set and administrative burden.

🧩 **Practical implications of delegation**
  
  - Delegating subdomains into their own zones typically requires additional authoritative servers: one for the parent zone and one for each delegated subzone (example results in **four authoritative name servers** total).  
  - Delegation enables independent control of subdomains and scales administration across teams or locations.

🛠 **Zone files and required records**
  
  - Zones are configured via **zone files**, text configuration files that declare all resource records for that zone.  
  - A zone file must contain an **SOA (Start of Authority)** record that **declares the zone** and names the primary authoritative server.  
  - Zone files usually include **NS** records listing other name servers responsible for the zone (supporting multiple physical servers).  
  - Zone files commonly contain other RR types: **A**, **AAAA** (IPv6), **CNAME**, plus configuration items such as a default **TTL** for records.

🔁 **Redundancy and multiple authoritative servers**
  
  - Although discussions often use the singular “server,” production setups usually use **multiple physical authoritative servers** (each with its own FQDN and IP).  
  - Multiple servers provide **fault tolerance**: if one server fails, others continue serving DNS for the zone.

↕️ **Depth and limits of zone hierarchies**
  
  - Zones can be nested multiple layers deep just like subdomains, but in practice **zones rarely extend many levels**; shallow delegation is far more common.

🔍 **Reverse lookup zones**
  
  - **Reverse lookup zone files** map an IP address back to an FQDN instead of names to IPs.  
  - Reverse zones use **PTR (pointer)** records (rather than A/AAAA) and otherwise follow the same zone-file conventions.
