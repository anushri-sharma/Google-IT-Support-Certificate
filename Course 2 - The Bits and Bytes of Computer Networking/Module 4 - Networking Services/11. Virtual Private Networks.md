
# [Virtual Private Networks](https://www.youtube.com/watch?v=4nr4P0nLI5s&t=4s)

🔒 Businesses' motivations and existing protections  
  – **Companies protect networks** to keep proprietary information and internal services accessible only to authorized users.  
  – Common protections include **firewalls**, **NAT**, and **non-routable address space** to limit access to devices physically on the LAN.  
  – These protections create a problem when employees are **off-site** (working from home or traveling) but still need internal resources.

💻 What a VPN is and primary use cases  
  – **VPN (Virtual Private Network)**: a technology that extends a private/local network to hosts that are not physically on that network.  
  – Most common use: **remote employee access**—allows a laptop outside the office to act like it's on the corporate LAN.  
  – Another common use: **site-to-site** connectivity—routers or specialized VPN devices link two separate office networks so they can act as one network.

🛠 How VPNs provide access (tunneling and virtual interfaces)  
  – VPNs are a **tunneling protocol**: they provision access to network resources that are not locally available by carrying one packet set inside another.  
  – Establishing a VPN often creates a **virtual interface** on the client with an IP from the remote network's address space.  
  – Sending data via that virtual interface lets the remote host access internal resources **as if physically connected** to the private network.

🔐 Packet handling and encryption mechanics  
  – Most VPNs place an **encrypted payload** into the transport-layer payload; that encrypted payload contains a full second set of packets (network, transport, and application layers) intended for the remote network.  
  – At the VPN endpoint the outer layers are removed, the payload is **decrypted**, producing the inner packet's top layers which are then re-encapsulated with appropriate data-link information and forwarded.  
  – The same encapsulation/decapsulation process occurs in reverse for traffic returning across the tunnel.

🧾 Authentication and security practices  
  – VPNs typically enforce **strict authentication** so only authorized users/devices can connect.  
  – **Two-factor authentication** became common with VPNs: a short-lived numeric token generated by hardware or software is required in addition to username/password.

🔗 Site-to-site specifics and deployment variants  
  – **Site-to-site VPNs**: a router or VPN appliance on one network establishes the tunnel to a peer router/appliance on another network, making two physically separated offices act like one network.  
  – Functionally similar to remote-user VPNs, but endpoints are network devices rather than individual clients.

🧩 Variability and design implications  
  – **VPN is a general concept, not a single protocol**—many implementations exist and details can differ widely.  
  – Security posture depends on implementation choices (encryption, authentication methods, endpoint configuration) and should be evaluated per deployment.  

✨ Core takeaway (technical effect)  
  – **VPNs use encrypted tunnels** to make a remote computer or network behave as if it is connected to a network it is not physically connected to, enabling secure remote and inter-site access.
