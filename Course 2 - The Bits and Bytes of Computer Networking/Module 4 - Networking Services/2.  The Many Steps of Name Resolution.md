
# [The Many Steps of Name Resolution](https://www.youtube.com/watch?v=ZLEOQxlhn6M&t=1s)

🔥 DNS purpose and name resolution  
  - Converts **domain names** into **IP addresses** so humans can use memorable names while computers use numeric addresses.  
  - The process of turning a domain name into an IP address is called **name resolution**.

💡 Standard host network configuration (four items)  
  - Hosts have a hard-coded **MAC address** tied to hardware.  
  - The remaining **four** standard configurable items are **IP address**, **subnet mask**, **gateway**, and **DNS server**.  
  - A host can operate without DNS, but lack of DNS makes human use impractical.

📂 Five primary DNS server roles (one server can play multiple roles)  
  - **Caching name servers**  
    - Store recent domain lookups to avoid repeating full resolution steps.  
  - **Recursive name servers**  
    - Perform full, step-by-step DNS resolution when no cached answer exists.  
  - **Root name servers**  
    - The entry point for full resolution; respond with the appropriate **TLD** name server to query.  
  - **TLD (top-level domain) name servers**  
    - Responsible for the last label of a domain (e.g., **.com**) and point to authoritative servers for domains under that TLD.  
  - **Authoritative name servers**  
    - Hold the definitive records for a domain (the last two labels, e.g., **weather.com**) and return the final IP address.

🍳 Caching + recursion behavior and typical deployment  
  - ISPs or local networks usually provide a name server that is both **caching** and **recursive**.  
  - Caching prevents repeating the entire multi-step resolution for every new TCP connection.  
  - A local host also typically maintains a small **temporary DNS cache** to reduce queries to the local name server.

🧭 Example: how caching avoids repeated full lookups (www.facebook.com)  
  - Friend queries **www.facebook.com**; both hosts use the same local name server.  
  - Local server does a full recursive resolution, returns the IP to friend, and **stores it in cache**.  
  - When you query **www.facebook.com** shortly after, the local server returns the cached IP without repeating the full resolution.

⏱️ TTL (Time To Live) for DNS records  
  - **TTL** is a configurable value in **seconds** set by the domain owner that dictates how long name servers may cache a record.  
  - Historically TTLs were very long (a day or more) to save bandwidth; today many TTLs are **minutes to hours**, but some long TTLs still exist.  
  - Changes to DNS records may not be globally visible until the previous TTL expires, so propagation delay can be up to the length of the TTL.

🌐 Full recursive resolution path (step-by-step)  
  - Step 1: Local recursive server contacts a **root name server**.  
    - There are **13** root authorities (13 root server IPs) that are globally served via **anycast**, meaning many physical instances may answer but they advertise those 13 addresses.  
    - Anycast routes a request to one of many physical destinations for the same IP based on location, congestion, or routing.  
  - Step 2: Root server returns the appropriate **TLD** name server to query (e.g., .com).  
  - Step 3: TLD name server returns the **authoritative name server** for the domain (e.g., weather.com’s authoritative server).  
  - Step 4: Authoritative name server returns the **actual IP address** for the target host (e.g., www.weather.com).

🔒 Hierarchical design and security implications  
  - The strict, delegated hierarchy (root → TLD → authoritative) improves stability and helps prevent malicious redirection.  
  - Clients **trust** the IP address they receive and will blindly send traffic to it; hierarchical control by trusted entities reduces the risk of incorrect or malicious responses.

🔁 Practical performance implications  
  - Caching at ISP/local servers and at the client reduces latency and network load by avoiding repeated traversal of the full resolution path for each connection.
