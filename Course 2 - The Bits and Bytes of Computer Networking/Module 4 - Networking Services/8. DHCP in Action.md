
# [DHCP in Action](https://www.youtube.com/watch?v=yIfGxkAfPLg&t=4s)

🔌 DHCP overview: application-layer protocol that configures the network layer using lower layers to communicate when no IP is yet assigned.  
  - **DHCP** relies on the transport, network, data link, and physical layers to operate while its purpose is to provide **network-layer configuration**.  

<img width="1280" height="410" alt="image" src="https://github.com/user-attachments/assets/a835a180-e1b1-4765-aa33-ca389ace6198" />


🔁 DHCP discovery = four-step exchange: **Discover → Offer → Request → ACK**.  
  - **Discover**: client announces need for configuration.  
  - **Offer**: server proposes an address and options.  
  - **Request**: client accepts a specific offer.  
  - **ACK** (acknowledgment): server confirms assignment and supplies final configuration.  

📡 Message addressing and broadcasts used to operate without a configured IP.  
  - **Discover** message: encapsulated so the UDP datagram is **src port 68 → dst port 67**, IP datagram is **src IP 0.0.0.0 → dst IP 255.255.255.255**, delivered as a broadcast to all LAN nodes.  
  
<img width="1280" height="529" alt="image" src="https://github.com/user-attachments/assets/4501624a-7eb7-4548-b701-e3ef6a2fd16f" />

  - **Offer** message: server responds with **src port 67 → dst port 68**, IP **src = server actual IP → dst = 255.255.255.255** (broadcast).

🧾 Ports and encapsulation specifics.  
  - **DHCP listens on UDP port 67** (server) and client uses **UDP port 68**; discovery messages always originate from port **68**.  
  - DHCP messages are encapsulated in a UDP datagram which is encapsulated in an IP datagram (hence operable before client has an IP).  

🧭 How client and server identify intended recipient without IPs: **MAC address** fields.  
  - DHCP offer and ACK include the **client's MAC address** in a message field so the original client recognizes messages intended for it despite broadcasts.  

⚙️ Server decision and allocation modes.  
  - Server checks its configuration and chooses an IP based on allocation mode: **dynamic**, **automatic**, or **fixed** address allocation.  
  - **Dynamic**: temporary assignments from a pool; **automatic**: server assigns the same address if available; **fixed**: static mapping (MAC→IP).  

⚠️ Edge cases and policy behaviors.  
  - Multiple DHCP servers can respond to a single discover; client typically accepts the first/selected offer but could be configured to **reject offers outside a desired range** (rare).  
  - Offers are broadcast and reachable by all hosts; the offer is targeted to a client via the included MAC address, not by unicast IP.  

⏳ DHCP lease lifecycle: expiration, renewal, and release.  
  - Assignment is a **DHCP lease** that includes an **expiration time** (can be hours, days, or short).  
  - When a lease expires the client must repeat the full DHCP discovery process to obtain a new lease.  
  - A client can explicitly **release** its lease (e.g., on disconnect) to return the address to the server's pool.  

🔧 Result for the client: full network-layer configuration once ACK is received.  
  - After the **ACK**, the client's networking stack applies the provided configuration (IP, subnet, gateway, DNS, etc.) and can operate normally on the network.
