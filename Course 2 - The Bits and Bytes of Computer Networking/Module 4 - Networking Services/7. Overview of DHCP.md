🔥 **Core host configuration requirements on a TCP/IP network**  
  
  – **IP address**, **subnet mask**, **primary gateway**, and **DNS (name) server** are required for every host.  
  – **Subnet mask**, **primary gateway**, and **DNS server** are typically the same across most nodes; the **IP address** must be unique per device.  
  – Manually configuring these four items on hundreds of machines is time-consuming and error-prone.

💡 **What DHCP is and where it operates**  
  
  – **DHCP (Dynamic Host Configuration Protocol)** is an **application-layer protocol** that automates assigning network configuration to hosts.  
  – A client queries a **DHCP server** when it connects and receives its network settings (IP, mask, gateway, DNS, etc.) in one transaction.  
  – DHCP is essential knowledge for IT support and network troubleshooting because it centralizes and controls host addressing.

🍳 **Primary benefits and operational implications**  
  
  – Reduces administrative overhead by eliminating per-device manual configuration.  
  – Solves IP assignment complexity: clients only need *any* valid IP on the correct network, not a predetermined one.  
  – Preserves the need for **static IPs** on infrastructure devices (e.g., gateway routers, DNS servers) so administrators can reliably reach them for diagnostics if services fail.  
  – Using DHCP pools prevents maintaining an exhaustive list of every node and its IP.

🧭 **DHCP allocation methods — behaviors and consequences**  

<img width="1279" height="468" alt="image" src="https://github.com/user-attachments/assets/aece77dc-7667-43f4-a26a-e86f1e9ce93c" />

  – **Dynamic allocation**  
    – A range (pool) of IPs is reserved for clients and the DHCP server issues an available IP on request.  
    – A client's IP **can change frequently**, potentially on each reconnection.

  – **Automatic allocation**  
    – A pool is used but the DHCP server **remembers past assignments** and will **reassign the same IP to the same device when possible**.  
    – Provides quasi-persistence without manual static mapping.

  – **Fixed allocation (static mapping)**  
    – Administrators create a table mapping **MAC address → specific IP**; DHCP assigns that IP when the MAC is seen.  
    – If a MAC is not found, the server may **fallback** to automatic/dynamic assignment or **refuse** assignment.  
    – Can be used as a **security/control measure** to restrict network access to preapproved devices.

🔧 **Operational edge cases and troubleshooting notes**  
  
  – Clients that rely on DHCP may receive different IPs over time (dynamic), which can complicate logging, access control, and remote troubleshooting unless reservations or static mappings are used.  
  – If infrastructure (e.g., DNS) lacks a static IP and malfunctions, administrators may be unable to reliably reach it for repair.  
  – Fixed mappings improve control but increase administrative work and require keeping MAC lists accurate.

📦 **DHCP options beyond basic addressing**  
  
  – DHCP can deliver additional configuration items (DHCP options) beyond **IP, mask, gateway, DNS**, such as **NTP servers** (Network Time Protocol) and other service endpoints.  
  – Centralizing these options via DHCP ensures consistent configuration across clients and simplifies updates.

🧾 **Practical deployment pattern**  
  
  – Reserve static IPs for servers and critical network devices.  
  – Configure a DHCP pool for client devices (desktops, laptops, phones).  
  – Use automatic allocation or reservations for devices that need more persistent addresses without fully manual static configuration.
