
# [DNS and UDP](https://www.youtube.com/watch?v=EKOtyMZRfrg&t=2s)

🔥 **Why DNS commonly uses UDP (transport choice)**  
  – **UDP is connectionless**, so there is no setup or teardown traffic; this reduces overall packet overhead for each lookup.  
  – Most DNS requests and responses **fit inside a single UDP datagram**, making UDP an appropriate lightweight transport, ideal candidate for a connectionless protocol.
  – DNS queries are extremely frequent; minimizing per-lookup overhead matters because DNS lookups are a precursor to most subsequent network traffic.  
  – Local caches (on hosts and caching name servers) reduce load, but full recursive resolutions still occur and are sensitive to transport overhead.
  _  DNS is a great example of an application layer service that uses UDP for the transport layer instead of TCP.
  _  It's also worth calling out that DNS can generate a lot of traffic.
  _  It's true that caches of DNS entries are stored both on local machines and caching name servers. But it's also true that if the full resolution needs to be processed, we're talking about a lot more traffic.

#### Let's see what it would look like for a full DNS lookup to take place via TCP.

🍳 **TCP example: full recursive DNS lookup packet-count (step-by-step, minimum)**  
  – Host → 1. The host that's making the DNS resolution request would send a SYN packet to the local name server on port 53, which is the port that DNS listens on.
  2. This name server would then need to respond with a SYN-ACK packet.
  3. That means the original host would have to respond with an ACK in order to complete the three-way handshake.: TCP three-way handshake (SYN, SYN-ACK, ACK) = **3 packets**. 
  – Host → local server: DNS request; server → host: ACK = **2 packets** (now **5 total**).  
  ```
Now that the connection has been established, the original host would have to send the actual request.

I'd like the IP address for foo.com, please.

When it receives this request, the name server would have to respond with another ACK.

I got your request for food.com.

We're up to five packets sent now.

In our scenario, the first caching name server doesn't have anything cached for foo.com.

It needs to talk to a root name server to find out who's responsible for the.com TLD.

This would require a three-way handshake : the actual request, the ACK of the request,
the response, and then the ACK of the response.

Finally, the connection would have to be closed via a four-way handshake.

That's 11 more packets or 16 total.

Now that the recursive name server has the correct TLD name server, it needs to repeat that entire process to discover the proper authoritative name server.

That's 11 more packets, bringing us up to 27 so far.

Finally, the recursive name server would have to repeat the entire process one more time

while talking to the authoritative name server in order to actually get the IP of foo.com.

This is 11 more packets for a running total of 38.

Now that the local name server finally has the IP address of foo.com, it can finally
respond to the initial request.

It responds to the DNS resolver that originally made the request, and then this
computer sends an ACK back to confirm that it received the response.

That's two more packets, putting us at 40.

Finally, the TCP connection needs to be closed via a four-way handshake.

This brings us to a grand total of 44 packets at the minimum in order for a fully
recursive DNS request to be fulfilled via TCP.

Forty-four packets isn't really a huge number in terms of how fast modern networks operate,
but it adds up fast, as you can see.

Remember that DNS traffic is just a precursor to actual traffic.

A computer almost always performs a DNS lookup because it needs to know the IP
of a domain name in order to send it additional data, not just because it's curious.
```

  – Recursive name server → root server: connect/handshake + request/ACK + response/ACK + close (four-way) = **11 packets** (now **16 total**).  
  – Recursive → TLD server: repeat full TCP exchange = **11 packets** (now **27 total**).  
  – Recursive → authoritative server: repeat full TCP exchange = **11 packets** (now **38 total**).  
  – Recursive server → original resolver: response + ACK = **2 packets** (now **40 total**).  
  – Final TCP connection close (four-way handshake.) for the original resolver connection = **4 packets** (final **44 packets**).  
  – **Result:** a minimum of **44 packets** for a fully recursive DNS resolution using TCP.

💡 **UDP example: full recursive DNS lookup packet-count (step-by-step, minimum)**  
  – Host → The original computer sends a UDP packet to its local name server on port 53, asking for the IP for     foo.com: single UDP query = **1 packet**.  
  
  – Recursive → root server and back: query + response = **2 packets** (now **3 total**).  

```

The local name server acts as a recursive server and sends up a UDP packet to the root server,
which sends a response containing the proper TLD name server.

That's three packets.

```
  
  – Recursive → TLD server and back: query + response = **2 packets** (now **5 total**).  
```
The recursive name server sends a packet to the TLD server and receives back a response
containing the correct authoritative server.

We're now at five packets.
```

  – Recursive → authoritative server and back: query + response = **2 packets** (now **7 total**).  

```
Next, the recursive name server sends its final request to the authoritative name
server, which sends a response containing the IP for foo.com.

That's seven packets.
```

  – Recursive server → original resolver: single UDP response = **1 packet** (final **8 packets**).  

```
Finally, the local name server responds to the DNS resolver that made the request
in the first place with the IP for foo.com.

That brings us to a grand total of eight packets.
```
  – **Result:** a minimum of **8 packets**, demonstrating far lower overhead than TCP for the same resolution.

```
See, way less packets.

You can see now how much overhead TCP really requires, and for something as
simple as DNS, it's just not needed.

It's the perfect example for why protocols like UDP exist in addition to the
more robust TCP.

🔁 **How DNS handles reliability without UDP’s built‑in recovery**  
  – UDP provides no retransmission, ordering, or connection guarantees; DNS moves that responsibility to the application layer.  
  – A DNS resolver simply **retries the query** if no response is received (timeout and retry logic).  
  – DNS servers focus only on responding to incoming lookups; resolvers handle retries and recursion.

⚠️ **When DNS uses TCP (exceptions and edge cases)**  
  – **DNS over TCP exists and is widely used** when a response cannot fit in a single UDP datagram.  
  – When a UDP response is **too large** (truncated), the server signals this and the client retries over TCP to obtain the full response.  
  – Increasing protocol complexity and larger DNS payloads (DNSSEC, large records) make TCP fallback more common.

📌 **Other practical details and implications**  
  – DNS listens on **port 53** for both UDP and TCP.  
  – The TCP example illustrates how connection establishment, ACKs, and orderly teardown significantly increase packet count and latency compared to UDP.  
  – The packet-count difference matters at scale because DNS lookup overhead multiplies across many clients and many lookups.

🔬 **Summary of trade-offs (concise)**  
  – **UDP:** lower overhead, simpler, good for most DNS lookups; reliability handled by retries at the application layer.  
  – **TCP:** higher overhead due to handshake/teardown and ACKs, used when responses exceed UDP size limits or when TCP-specific features are required.

```
The DNS resolver just asks again if it doesn't get a response.

Basically, the same functionality that TCP provides at the transport layer is
provided by DNS at the application layer in the most simple manner.

A DNS server never needs to care about doing anything but responding to incoming lookups.

A DNS resolver simply needs to perform lookups and repeat them if they don't succeed, a real
showcase of the simplicity of both DNS and UDP.

I should call out that DNS over TCP does, in fact, exist and is also in use all over.

As the web has gotten more complex, it's no longer the case that all DNS lookup responses
can fit in a single UDP datagram.

In these situations, a DNS name server would respond with a packet explaining that the response is too large.

The DNS client would then establish a TCP connection in order to perform the lookup.
