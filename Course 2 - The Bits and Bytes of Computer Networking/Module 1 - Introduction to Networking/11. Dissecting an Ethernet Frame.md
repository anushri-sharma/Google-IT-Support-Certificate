
# [Dissecting an Ethernet Frame](https://www.youtube.com/watch?v=E1Au6a3o-6g&t=3s)


**A data packet is an all encompassing term that represents any single set of binary data being sent across a network link.**

<img width="1280" height="392" alt="image" src="https://github.com/user-attachments/assets/2ffc23a4-b203-4d5a-86a6-091406714ef4" />


The term **data packet** isn't tied to any specific layer or technology.

It just **represents a concept**, **one set of data being sent from point A to point B**. 

#### Data packets at the Ethernet level are known as Ethernet frames.

An **Ethernet frame** is a **highly structured collection** **of information presented in a specific order**.

This way, **network interfaces at the physical layer** can **convert** a stream of **bits** traveling across a link into meaningful data, or vice versa.

<img width="1278" height="374" alt="image" src="https://github.com/user-attachments/assets/4c8bf53d-37b9-4a0f-8ebe-b8a0b44a942c" />

Almost all sections of an Ethernet frame are mandatory, and **most of them have a fixed size.**

1.  The **first part** of an Ethernet frame is known as the **preamble**.

- **A preamble is 8 bytes, or 64 bits long,** and **can itself be split into two sections.**

- The first seven bytes are a series of alternating ones and zeros.

    - These act partially as a buffer between frames and can also be used by the network
interfaces to synchronize internal clocks they use to regulate the speed at which they send data.
<img width="1280" height="369" alt="image" src="https://github.com/user-attachments/assets/ede8593e-0eef-40e7-9c50-88cf80a582a6" />

- This last byte in the preamble is known as the **SFD** or **Start frame delimiter**.
<img width="1280" height="385" alt="image" src="https://github.com/user-attachments/assets/508df5b1-ac78-4c33-97e4-5364b013a821" />

<img width="1279" height="453" alt="image" src="https://github.com/user-attachments/assets/27b23428-5922-46c0-8fdb-37a828f4be52" />

   - This signals to a receiving device that the preamble is over and that the actual frame contents will now follow.

2. Immediately following the Start frame delimiter comes the **destination Mac address**.


<img width="1280" height="387" alt="image" src="https://github.com/user-attachments/assets/3987a156-e614-461a-ae03-0139254e756d" />


3. This is the hardware address of the intended recipient, which is then followed by the **source Mac address** or where the frame originated from.
   
<img width="1280" height="370" alt="image" src="https://github.com/user-attachments/assets/b9fcd253-a2aa-4b97-a9d6-50227e299e1a" />


Don't forget that each **Mac address is 48 bits, or 6 bytes long**.

4. The next part of an Ethernet frame is called the **ether-type field**.

<img width="1279" height="343" alt="image" src="https://github.com/user-attachments/assets/327920ac-a474-4a91-aca4-9e4659b7622b" />

It's **16 bits long** and used to describe the protocol of the contents of the frame.

<img width="1278" height="392" alt="image" src="https://github.com/user-attachments/assets/3cbc0859-a06f-4ea3-86f1-38d0a7bb7268" />

It's worth calling out that instead of the **ETHER type field**, you could also find what's known as a **VLAN header**.

5. It indicates that the frame itself is what's called a **VLAN frame**.

<img width="1280" height="408" alt="image" src="https://github.com/user-attachments/assets/46864842-d4a3-44e7-a3fc-bc7d3727e990" />

<img width="1257" height="373" alt="image" src="https://github.com/user-attachments/assets/46a638e7-92d3-476a-8d65-024e94a83436" />

If a VLAN header is present, the ETHERTYPE field follows it.
<img width="1278" height="299" alt="image" src="https://github.com/user-attachments/assets/91b8816d-916b-408f-940c-dc1ec7f6feae" />

**VLAN stands for virtual LAN.**
<img width="1274" height="606" alt="image" src="https://github.com/user-attachments/assets/18a97227-96e8-4be1-ad57-edc76aaa1b09" />

It's a technique that lets you have multiple logical LANs operating on the same physical equipment.

Any frame with a VLAN tag will only be delivered out of a switch interface configured to relay that specific tag.

This way you can have a single physical network that operates like it's multiple LANs.

**VLANs are usually used to segregate different forms of traffic.**
<img width="1280" height="357" alt="image" src="https://github.com/user-attachments/assets/3ca8b61b-d0af-4ce8-a0ef-acf613c62c24" />

So you might see a company's IP phones operating on one VLAN, while all desktops operate on another.

After this, you'll find the data payload of an ethernet frame.

6. A **payload**, in networking terms, is the actual data being transported, which is everything that isn't a header.
<img width="1278" height="340" alt="image" src="https://github.com/user-attachments/assets/c3c52744-4346-47ce-a269-65527d31a170" />
<img width="1271" height="391" alt="image" src="https://github.com/user-attachments/assets/a2e78709-d3a9-4f34-92b9-e5db404492ab" />

The **data payload** of a traditional ethernet frame can be anywhere from 46 to 1500 bytes long.

This contains all of the data from higher layers, such as the ip, transport and application layers, that's actually being transmitted.

7. Following that data, we have what's known as a **frame check sequence**.

<img width="1279" height="327" alt="image" src="https://github.com/user-attachments/assets/bd420b7f-4590-498e-b3ea-329ccd61133f" />
<img width="1279" height="379" alt="image" src="https://github.com/user-attachments/assets/b300741f-516f-427d-a582-15ecd31dc3e1" />

This is a 4 byte or 32 bit number that represents a checksum value for the entire frame.

This checksum value is calculated by performing what's known as a **cyclical redundancy check** against the frame.
<img width="1277" height="403" alt="image" src="https://github.com/user-attachments/assets/e973ef79-0614-4f0c-8e0d-fb406bfa1cb4" />

A **cyclical redundancy check**, or CRC, is an important concept for data integrity and is used all over computing, not just network transmissions.

- A **CRC is basically a mathematical transformation that uses polynomial division** to **create a number that represents a larger set of data**.

- **Anytime you perform a CRC** against a set of data, you should **end up with the same checksum number.**

- The reason it's included in an ethernet frame is so that the receiving network interface can infer if it received uncorrupted data.

- When a device gets ready to send an ethernet frame, it collects all the information

- we just covered, like the destination and originating Mac, addresses the data payload, and so on.

- Then it performs a CRC against that data and attaches the resulting checksum number as the frame checks sequence at the end of the frame.

This data is then sent across a link and received at the other end.
<img width="1280" height="568" alt="image" src="https://github.com/user-attachments/assets/20802544-e992-45ee-9664-7a7c93776c49" />
<img width="1174" height="561" alt="image" src="https://github.com/user-attachments/assets/e23c39c2-f4db-4612-8fea-79b16c62106d" />

Here, all the various fields of the Ethernet frame are collected, and now the receiving side performs a CRC against that data.

If the **checksum computed** by the **receiving end doesn't match the checksum** in the frame check sequence field, then the **data is thrown out**.

This is **because** some amount of **data** must have been **lost or corrupted during transmission.**

It's then up to a protocol at a higher layer to decide if that data should be retransmitted.

**Ethernet itself only reports on data integrity.
It doesn't perform data recovery.**
