
# [Routing Tables](https://www.youtube.com/watch?v=j-7yCGIHkbY&t=16s)


🔥 **What a routing table is and historical context**  
  - **Routing table**: a data structure routers consult before transmitting packets to decide where to send them next.  
  - Early routers were ordinary computers with two network interfaces that bridged networks and used **manually updated routing tables**.  
  - **Modern OSes still maintain a routing table** and consult it for every outbound packet.  
  - You can build a simple router today with a computer, two NICs, and a manually configured routing table.

🍳 **Common overall behavior and role**  
  - When a router receives a packet it **examines the destination IP** and matches it to a network entry in the routing table.  
  - If no explicit match exists, the router uses a **catch-all (default) entry** that matches any IP not otherwise listed.  
  - Routing tables can be **static (manual)** or populated/updated from other routers (dynamic), though specifics of routing protocols are separate from the table structure.

💡 **Column: Destination network (definition)**  
  - Contains a row for **each network the router knows about**—the network identifier plus netmask.  
  - Network entries may be stored as a single **CIDR-style** field (e.g., 192.0.2.0/24) or split into **network ID** and **netmask** columns; both represent the same concept.  
  - The router uses this definition to determine **which IP addresses belong to that network**.

**Destination network.**

This column would contain a row for each network that the router knows about.

This is just the definition of the remote network, a network ID and a net mask.

These could be stored in one column insider notation, or the network ID and net mask might be in a separate column.

Either way, it's the same concept.

The router has a definition for a network and therefore knows what IP addresses might live on that network.

When the router receives an incoming packet, it examines the destination IP address and determines which network it belongs to.

A routing table will generally have a catch-all entry that matches any IP address that it doesn't have an explicit network listing for.

📍 **Column: Next hop (where to send packets next)**  
  - Specifies the **IP address of the next router** to receive traffic bound for the destination network.  
  - Can alternatively indicate the network is **directly connected**, meaning **no additional hops** are required beyond the router itself.

📏 **Column: Total hops / distance metric (how far the destination is)**  
  - Stores a **distance metric** (commonly hop count or other metrics) that represents how far or costly a route is.
  - Routers use this to **choose the shortest/best path** when multiple paths to the same destination exist.
  
<img width="1280" height="616" alt="image" src="https://github.com/user-attachments/assets/0aaaadfa-5299-4f30-9e10-566266fa7ca5" />

  - Distance is dynamic: **intermediate routers can fail, links can disconnect, new routers can appear, and congestion can change effective route quality**, so metrics must be updated.  
  - Routers compare metrics from neighboring routers to determine whether a newly learned path is **better or worse** than the current one.

🔌 **Column: Interface (egress port to use)**  
  - Identifies **which local interface** the router should forward traffic out of for that destination network.  
  - Ties the logical route entry to a physical or virtual export path on the router.

⚙️ **Scale and operational implication**  
  - Routing tables range from **simple** (a few entries on a home router) to **millions of rows** on core Internet routers.  
  - Every packet traversing a router requires a routing-table lookup, so large tables imply important **performance and lookup-efficiency** requirements.  
  - Maintaining accurate distances/metrics and interfaces at scale requires mechanisms beyond manual editing (i.e., routing protocols and optimized lookup implementations).
