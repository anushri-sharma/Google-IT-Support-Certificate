
# [IPv4 Datagram and Encapsulation](https://www.youtube.com/watch?v=oIhc3hbivBg&t=8s)

- Under the **IP protocol**, a packet is usually referred to as an **IP datagram**.

- Just like any ethernet frame, **IP datagram** is a highly structured series of fields that are strictly defined.

🔷 **IP datagram overview: structure and role**  
  – **IP datagram** = network-layer packet; consists of two main parts: **header** and **payload**.  
  – IP header contains substantially more structured fields than an Ethernet frame header.  
  – The IP datagram as a whole becomes the **payload of an Ethernet frame**; the IP payload typically contains an entire **TCP or UDP packet**.

🧾 **Version field**  
  – **4 bits** that indicate IP protocol version.  
  – Common values: **IPv4** (most common) and **IPv6** (increasing adoption).

<img width="1277" height="262" alt="image" src="https://github.com/user-attachments/assets/8a4070c2-648a-4d8c-bdbe-c86ce59cbe8c" />


📏 **Header Length field**  
  – **4 bits** that declare the total header length.  
  – For **IPv4** the header is almost always **20 bytes**; **20 bytes** is the **minimum** IPv4 header length.
  - You couldn't fit all the data you need for a properly formatted IP header in any less space.

🍳 **Service Type (Type of Service) field**  
  – **8 bits** used for QoS (quality of service) signaling.  
  – Routers can use this to prioritize some datagrams over others.

<img width="1276" height="392" alt="image" src="https://github.com/user-attachments/assets/a47fee69-6ee8-49c8-b386-96b868c5dcb3" />


📦 **Total Length field**  
  – **16 bits** indicating the total length of the IP datagram (header + payload).  
  – Implies a single-datagram maximum size of **65,535 bytes** (largest number representable in 16 bits).

<img width="1275" height="410" alt="image" src="https://github.com/user-attachments/assets/22453477-7c11-4ab2-a5ae-117fbf45b3d5" />


🆔 **Identification field**  
  – **16 bits** used to tag fragments that belong to the same original datagram.  
  – When data exceeds the per-datagram limit or an intervening network has a smaller allowed datagram size, the IP layer splits the data and assigns the same identification value to all resulting fragments to allow reassembly.

🧩 **Flags and Fragmentation Offset fields**  
  – **Flags** indicate whether fragmentation is allowed and whether a datagram is a fragment.  
  – **Fragmentation offset** provides positioning information so the receiver can reassemble fragments in the correct order.  
  – Fragmentation occurs when a datagram traverses from a network permitting larger datagrams to one permitting smaller datagrams.

⏱️ **Time To Live (TTL) field**  
  – **8 bits** specifying how many router hops a datagram may traverse before being discarded.  
  – Each router **decrements TTL by one**; when TTL reaches zero, the datagram is not forwarded.  
  – Purpose: prevent infinite routing loops (e.g., misconfigured mutual next-hop loops).
  - Once this **value reaches zero**, a router knows it **doesn't have to forward** the datagram any further.

🔗 **Protocol field**  
  – **8 bits** that identify the transport-layer protocol carried in the IP payload (common values: **TCP**, **UDP**).
  - The most common transport layer protocols are **TCP and UDP**.
<img width="1280" height="329" alt="image" src="https://github.com/user-attachments/assets/e1c16938-bff5-4ece-9479-d8cef489668b" />


✅ **Header Checksum field**  
  – Checksum over the IP **header only**; used to detect header corruption.  
  – Must be recomputed at each router because the **TTL** (and thus header contents) changes during transit.
  - This field is a check some of the contents of the entire IP datagram header.
  - It functions very much like the ethernet checksum field we discussed in the last module since the TTL field
    has to be re computed at every router that a data gram touches the check some field necessarily changes too.

📫 **Source and Destination IP address fields**  
  – Each address is **32 bits** in IPv4.  
  – Provide the network-layer endpoints for delivery.
  - IP address is a 32 bit number so it should come as no surprise that these fields are each 32 bits long

⚙️ **IP Options and Padding fields**  
  – **IP Options** are optional, variable-length fields used for special behaviors and primarily for testing. 
        - **IP options field** is both **optional and variable in length.**
  – **Padding** (zeros) follows options as needed so the header length aligns to the declared header size.

#### You might wonder how this relates to what we've learned so far.
You might remember that in our breakdown of an ethernet frame we mentioned a section we described as the data payload section.

This is exactly what the IP datagram is and this process is known as encapsulation.

🔁 **Encapsulation and layering implications**  
  – The **entire IP datagram** is placed inside an **Ethernet frame’s payload (encapsulation)**.  
    - The entire contents of an IP datagram are encapsulated as the payload of an ethernet frame.

<img width="1274" height="604" alt="image" src="https://github.com/user-attachments/assets/1f6df19f-4a97-4006-9d46-1d2104b94d09" />

  – The IP datagram itself encapsulates a transport-layer packet (TCP/UDP).  
  – Layering ensures each layer provides services required by the layer above and remains independent of the layer below.

  - **IP datagram also has a payload section.**
  - **The contents of this payload are the entirety of a TCP or UDP packet.**
