
# [Traceroute](https://www.youtube.com/watch?v=xazhVT8Hpew&t=23s)

🔍 Traceroute purpose and use  
  – **Traceroute** discovers the network path between two nodes and identifies where problems occur **along the chain of router hops**.  
  – It provides per-hop visibility and latency information so you can pinpoint which intermediary hop is adding delay or failing.

🧭 TTL-based mechanism (how traceroute maps hops)  
  – Traceroute manipulates the **TTL (Time To Live)** field in IP packets: each router decrements TTL by **1**, and when TTL reaches **0** the router discards the packet and returns an **ICMP time-exceeded** message to the origin.  
  – Traceroute sends successive packets with TTL = **1**, **2**, **3**, … so the first packet expires at the first router, the second at the second router, etc., producing one ICMP reply per hop until the destination is reached.

⏱️ Per-hop probing and timing details  
  – For **each hop**, traceroute sends **three identical probes** (like ping) and records the **round-trip time (RTT)** for each.  
  – This yields three RTT values per hop that reveal transient vs consistent latency and packet loss patterns at that specific hop.

📋 Traceroute output contents  
  – Each line corresponds to a hop and shows the **hop number**, the **RTTs for the three probes**, the **IP address** of the hop, and a **host name** if reverse DNS resolves it.  
  – Traceroute continues incrementing TTL until a probe reaches the **destination**, at which point the path is complete.

🖥️ Platform differences and defaults  
  – On **Linux/macOS**, traceroute typically sends **UDP packets to very high port numbers**.

⚙️ Related long-running tools and aggregate behavior  

**Two more tools that are similar to traceroute are:**

  – **mtr on (Linux/macOS)** runs continuously, **updates in real-time**, and shows ongoing aggregated statistics for each hop so you can observe changes over time.  
  – **pathping on (Windows)** runs for **50 seconds** and then presents final aggregated statistics for the traced path.

🔁 Practical implications and edge cases  
  – Because traceroute relies on ICMP/UDP behavior, results can be affected if intermediate routers or hosts **block or deprioritize ICMP/UDP**, producing timeouts or missing hostnames even though connectivity exists.  
  – Per-hop RTTs reflect the round-trip to each hop’s responding interface, not necessarily end-to-end application performance; asymmetric routing or ICMP rate-limiting can skew interpretation.
