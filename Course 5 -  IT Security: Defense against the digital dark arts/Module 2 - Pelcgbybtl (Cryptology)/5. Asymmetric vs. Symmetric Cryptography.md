
# [Asymmetric vs. Symmetric Cryptography](https://www.youtube.com/watch?v=Y5_BDExvuQE&t=8s)

🔐 **Asymmetric vs. Symmetric — core trade-offs**  
  – **Asymmetric encryption** enables secure communication over an **untrusted channel** because keys for encryption and verification are different (public/private), eliminating the need to pre-share a secret.  
  – **Symmetric encryption** requires a **shared secret/key** known to both parties beforehand, so secure key distribution is necessary.  
  – **Asymmetric is more computationally expensive and complex; symmetric is faster and far more efficient for encrypting large volumes of data.**

⚖️ **Common practical pattern: hybrid encryption**  
  – Use **asymmetric encryption for key exchange**: transmit a symmetric key securely to the other party using public-key operations.  
  – Use **symmetric encryption for bulk data**: once the shared secret is received, use a symmetric cipher to encrypt large messages quickly and efficiently.  
  – This leverages the **security of asymmetric methods** for bootstrapping and the **performance of symmetric methods** for throughput.

🔏 **Message Authentication Codes (MACs) — purpose and properties**  
  – A **MAC** authenticates a received message and ensures **integrity** (message was not modified) and **authenticity** (message came from a party holding the shared secret).  
  – MACs are **symmetric** primitives: the same secret key is used to both **generate** and **verify** the MAC, so the key must be **pre-shared or exchanged securely**.  
  – MACs differ from **digital signatures**, which use public-key pairs and allow verification without a shared secret.

🧩 **HMAC — keyed-hash message authentication code**  
  – **HMAC** combines a **cryptographic hash function** with a **secret key** to produce a MAC.  
  – Verification: receiver recomputes HMAC over the received message with the shared key and compares it to the transmitted MAC; a match authenticates the message.  
  – Security depends on the **underlying hash function**; older hashes like **MD5 are considered insecure** and should not be used; prefer modern hashes (for example, **SHA-256**).  
  – Common examples historically include **HMAC-SHA1**, but prefer stronger hash choices for new systems.

🔐 **CMAC and block-cipher–based MACs**  
  – **CMAC** (Cipher-based MAC) derives a MAC using a **symmetric block cipher** (e.g., **AES**) instead of a hash.  
  – Process: use the shared symmetric key to encrypt data and derive a MAC from the encryption output; verification recomputes similarly.  
  – CMACs are appropriate when you already rely on a block cipher and want a symmetric MAC primitive standardized for block ciphers.

🧱 **CBC-MAC — mechanism and caveats**  
  – **CBC-MAC** uses a block cipher in **CBC mode**: plaintext blocks are processed sequentially with each ciphertext block influencing the next via XOR chaining; the **final block output** serves as the MAC.  
  – Integrity property: any plaintext modification changes the final CBC output, enabling detection of tampering.  
  – **Security caveat**: plain CBC-MAC is secure only for **fixed-length messages**; it is vulnerable when used naively on variable-length messages. Use **CMAC** or other standardized constructions to avoid these pitfalls.

🔎 **Operational implications and best practices**  
  – Use **asymmetric methods** for secure key exchange and identity-establishment; avoid using them for encrypting large payloads due to cost.  
  – Use **symmetric encryption (AES, etc.)** for bulk data once a secure shared key exists.  
  – Use **HMAC (with a secure hash like SHA-256)** or **CMAC (with AES)** for message authentication depending on whether you prefer hash-based or cipher-based MACs.  
  – Avoid deprecated primitives (e.g., **MD5**) and avoid naive CBC-MAC on variable-length messages; follow standardized MAC constructions.
