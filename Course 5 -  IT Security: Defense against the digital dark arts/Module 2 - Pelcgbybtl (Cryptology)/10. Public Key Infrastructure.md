
 # [Public Key Infrastructure](https://www.youtube.com/watch?v=7TEJYa1a6Hk&t=5s)

 🔐 **Public Key Infrastructure (PKI)**  
  - **Definition:** System that defines the **creation, storage, and distribution of digital certificates** to secure communications and verify identities.  
  - **Purpose:** Enables trusting public keys for encrypted channels and digital signatures when certificate signatures validate and the signer is trusted.

🧾 **Digital certificate contents and verification**  
  - **Contents:** **Public key**, **identity information** (the entity it belongs to), and a **digital signature** from an authority that verified the identity.  
  - **Verification logic:** If the certificate signature is valid and the signing entity is trusted, the certificate’s public key can be trusted for secure communication.

🏛️ **Certificate Authority (CA) and Registration Authority (RA)**  
  - **CA:** Entity responsible for **issuing, storing, and signing certificates**; central to PKI trust.  
  - **RA:** Verifies identities of certificate requesters before CA issues certificates; often combined with the CA role in practice.  
  - **Infrastructure needs:** A **central repository** to store and index certificates/keys and a **certificate management system** to control issuance, storage access, and lifecycle.

<img width="1277" height="406" alt="image" src="https://github.com/user-attachments/assets/3ec4856e-5a9b-4bc3-9ff9-877e50458ad3" />

A digital certificate is a file that proves that an entity owns a certain public key. A certificate contains information about the public key,
the entity it belongs to, and a digital signature from another party that has verified this information.

If the signature is valid and we trust the entity that signed the certificate,
then we can trust the public key to be used to securely communicate with the entity that owns it. The entity that is responsible for storing, issuing,
and signing certificates is referred to as CA, or Certificate Authority. It's a crucial component of a PKI system.

🧩 **Common certificate types and behaviors**  
  - **SSL/TLS server certificates:** Presented by web servers during SSL/TLS setup.  
    - Client (e.g., browser) verifies the certificate **subject matches the server hostname** and that it is **signed by a trusted CA**.  
    - Certificates can be valid for **multiple hostnames** (SANs) and **wildcard certificates** use an **asterisk (*)** to cover all hostnames within a domain.  
    - **Self-signed certificates** are signed by their own private key; they **fail verification unless pre-trusted** by the client.  
  - **SSL/TLS client certificates:** Optional; authenticate clients to servers for access control.  
    - Typically **issued/managed by an internal CA**, not a public CA; less commonly used than server certs.  
  - **Code signing certificates:** Used to **sign executables**, allowing recipients to verify the **integrity** of the application and that it **originated from the claimed author** (protects against tampered or spoofed software).

🌳 **Chain of trust and certificate hierarchy**  
  - **Root certificates:** **Self-signed** because they are the trust anchor with no higher authority.  
  - **Intermediates (subordinate CAs):** Created when a root CA signs a certificate with the **CA field set to true (CA=true)**; these intermediates can **sign other certificates** and extend trust.  
  - **Tree structure:** Root private key at top → intermediates → further intermediates → **end-entity / leaf certificates** (no CA authority).  
  - **End-entity/leaf certificates:** Final nodes that are used by servers, clients, or code; they **cannot act as CAs**.

⚠️ **Bootstrapping trust and distribution of root CAs**  
  - **Trust bootstrap problem:** The chain is only valid if the root CA certificate itself is trusted out-of-band.  
  - **Distribution method:** Major **OS vendors ship trusted root CA certificates** with their operating systems; browsers typically use the **OS-provided root store**.  
  - **Implication:** If a root is not present in the trusted store, the entire chain that relies on it is **untrusted** to the client.
