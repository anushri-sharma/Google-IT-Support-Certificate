# Notes


The video begins by emphasizing the importance of cryptographic hash functions in **authentication systems**, particularly when users log into their accounts (e.g., email). The basic authentication mechanism involves verifying that the password a user enters matches the stored password for that account. However, **storing passwords in plain text is highly insecure** and should never be done.

<img width="1261" height="318" alt="image" src="https://github.com/user-attachments/assets/04cad4d4-2934-413e-9bf8-6199c416aed0" />

 
Instead of storing the actual passwords, modern systems store **hashes of the passwords**. When a user logs in, the entered password is hashed using the same function, and the resulting hash is compared to the stored hash. If the hashes match, the user is authenticated. This approach protects passwords because even if the system is compromised, attackers will only obtain the hashes, not the actual passwords.

 
Storing hashes instead of plain text passwords means that if an attacker gains access to the database, the damage is limited to stolen password hashes. However, attackers can still try to recover the original passwords by performing a **brute force attack** on the hashes. This involves systematically guessing passwords, hashing each guess, and checking if the hash matches any stolen hashes.

 
Brute force attacks can be very **computationally intensive**, depending on the hash function used. Importantly, brute force attacks can never be completely prevented because given **unlimited time and resources**, attackers can eventually guess any password. The defense strategy focuses on making brute force attempts **computationally expensive and time-consuming**, thus impractical.


One common method to increase computational difficulty is to use **hash iteration**, where the password is hashed multiple times (sometimes thousands of iterations). This significantly raises the cost of each guess attempt for an attacker, thereby improving security.

 
The video then introduces **rainbow tables**, which are precomputed tables of password-hash pairs used to speed up password recovery from stolen hashes. Rainbow tables trade off **computation time for storage space** by precomputing hashes for many possible passwords and storing them in a lookup table.

<img width="1277" height="586" alt="image" src="https://github.com/user-attachments/assets/ac62b535-7f22-4307-a796-dd6fa07ed0a2" />

 
Unlike brute force attacks, which compute hashes on the fly for each guess, rainbow tables allow attackers to simply look up the hash in their table to find the corresponding password, drastically reducing the time needed to recover passwords. These tables can be downloaded from the internet for widely used hashing functions and common password lists.


To defend against rainbow table attacks, systems use **password salts**, which are random data added to passwords before hashing. The salt is concatenated with the password, and the combination is hashed. The generated hash and the salt are stored together. This means the hash is unique not only to the password but also to the salt.

<img width="1280" height="417" alt="image" src="https://github.com/user-attachments/assets/af98c1a6-b0b6-4348-9e6a-5005df4bbbf3" />

With salts, an attacker must generate a unique rainbow table for every possible salt value, exponentially increasing the storage and computation required. This renders rainbow table attacks practically infeasible when using sufficiently large salts.

<img width="1265" height="608" alt="image" src="https://github.com/user-attachments/assets/69d1e806-308f-4b55-8abc-4e8c1d5b37d9" />

 
The examples of salt sizes used in different systems:

| System           | Salt Size (bits) | Number of Possible Salts           | Explanation                                  |
|------------------|------------------|----------------------------------|----------------------------------------------|
| Early UNIX       | 12               | $2^{12} = 4096$                  | Attacker must generate rainbow tables 4096 times over |
| Modern Linux/BSD/Solaris | 128              | $2^{128} \approx 3.4 \times 10^{38}$ | Over 340 undecillion salts, making rainbow tables infeasible |

The large salt size in modern systems effectively **prevents rainbow table attacks** within any realistic timeframe.

<img width="1268" height="295" alt="image" src="https://github.com/user-attachments/assets/497b6c21-5577-4372-bd42-48a01e56c429" />


The lesson concludes by highlighting how adding salt to password hashing dramatically improves security by making precomputed attacks infeasible, reinforcing the principle that **proper use of salts and iterative hashing is essential for protecting stored password hashes** against offline attacks.
