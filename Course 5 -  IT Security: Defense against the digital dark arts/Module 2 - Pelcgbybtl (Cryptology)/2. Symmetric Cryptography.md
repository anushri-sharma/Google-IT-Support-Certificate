# Notes

 
The video begins by emphasizing the importance of **decryption in cryptographic systems**, highlighting that encrypted messages must be decoded by the recipient to be useful. It introduces the first broad category of encryption algorithms: **symmetric key algorithms**. These use the **same key for both encryption and decryption**. The segment recalls **Kerckhoff’s principle**, underscoring that the **key—not the algorithm—must remain secret** to protect communications from eavesdroppers.

 <img width="1280" height="348" alt="image" src="https://github.com/user-attachments/assets/2888c785-db13-4a08-b527-4ec82adddf85" />

The focus is on **symmetric key encryption**, where **the key is shared and used identically for both encoding and decoding** messages. This approach assumes that both sender and receiver have access to the same secret key, making the security dependent on keeping this key confidential.

 
To explain these concepts, a **substitution cipher** is used as a simple example of symmetric encryption. In substitution ciphers, parts of the plaintext are replaced by ciphertext characters based on a **mapping table**, which acts as the key. Without this key, decoding the ciphertext is impractical. This example ties back to the earlier "Hello World" demonstration, where characters are substituted systematically.

<img width="1275" height="389" alt="image" src="https://github.com/user-attachments/assets/7e5332de-00ef-49ef-be93-5ca7b1a0632a" />

 
A well-known substitution cipher example is the **Caesar cipher**, which works by **shifting the alphabet by a fixed number of positions**. This shift value is the key. The cipher replaces each plaintext character with another character offset by the shift amount, creating the ciphertext. The key concept is that the key determines the substitution pattern.

<img width="1277" height="258" alt="image" src="https://github.com/user-attachments/assets/c332404a-68ae-4f35-bf9b-bca961e2b5e0" />
 
A specific Caesar cipher variant is **ROT13**, which shifts the alphabet by 13 places. ROT13 is effectively a Caesar cipher with a fixed key of 13. The video revisits the "Hello World" example encoded with ROT13, resulting in the ciphertext **"URYYB JBEYQ"**. Decoding is done by reversing the substitution using the same ROT13 operation, demonstrating the **symmetry of the cipher**.

<img width="1276" height="636" alt="image" src="https://github.com/user-attachments/assets/135d0925-d1c1-4886-ae80-287b234acb1e" />
 
The video highlights a unique property of ROT13: since the shift is exactly half the alphabet (13 out of 26), **ROT13 is its own inverse**. Applying ROT13 to ciphertext returns the original plaintext. This is a distinctive feature not shared by other Caesar shifts.

<img width="1274" height="639" alt="image" src="https://github.com/user-attachments/assets/f66e4e08-c523-4735-b39c-f711a2e8ca03" />

 
Following substitution ciphers, the video outlines two more categories of symmetric key ciphers based on their method of processing plaintext:

| Cipher Type    | Description                                                                                      | Operation Details                         |
|----------------|------------------------------------------------------------------------------------------------|------------------------------------------|
| Stream Cipher  | Encrypts data one character or digit at a time, producing one encrypted character per input.    | One-to-one input-output relationship.    |
| Block Cipher   | Processes fixed-size blocks of plaintext as whole units, encrypting each block independently.   | Padding used if plaintext < block size.  |

 
**Stream ciphers** encrypt data as a continuous stream, handling each unit individually. This often results in **faster and less complex implementations** but may be **less secure if key management is poor**. 

<img width="1277" height="459" alt="image" src="https://github.com/user-attachments/assets/1ce4dfca-3a29-4360-a95d-ca39ebcbeab1" />
 
**Block ciphers** group plaintext into fixed-size blocks and encrypt each block as a unit. If the plaintext is smaller than the block size, **padding is added** to fill the block. Block ciphers are generally more secure but potentially more computationally intensive.

<img width="1278" height="471" alt="image" src="https://github.com/user-attachments/assets/71701aec-1f41-4614-81da-879168ade6f3" />
 
A critical security consideration is the **reuse of encryption keys**. When the same key is used multiple times, the cipher’s security can be compromised, allowing attackers to potentially recover the plaintext.


To mitigate key reuse risks, **Initialization Vectors (IVs)** are introduced. An IV is a **random value combined with the master key** to generate a unique encryption key for each message. This approach ensures that even with the same master key, the encryption key differs per instance, enhancing security.

<img width="1280" height="574" alt="image" src="https://github.com/user-attachments/assets/07168c79-259b-40bc-8953-705c6dd7b63f" />
 
The IV must be sent to the receiver along with the encrypted message in **plaintext form** so the decryption process can reconstruct the encryption key. The video references this practical use in **802.11 wireless frames**, where the IV is included in plaintext immediately before the encrypted data payload in web-encrypted wireless packets.

### Key Insights and Concepts

- **Symmetric key algorithms** use a single secret key for both encryption and decryption.
- The **key—not the algorithm—is the secret** as per **Kerckhoff’s principle**.
- **Substitution ciphers** replace plaintext characters with ciphertext characters based on a key mapping.
- The **Caesar cipher** shifts alphabets by a fixed offset; the offset number is the key.
- **ROT13** is a special Caesar cipher with a 13-character shift and is its own inverse.
- Symmetric ciphers are categorized as:
  - **Stream ciphers** (encrypt one character/digit at a time).
  - **Block ciphers** (encrypt fixed-size blocks of data).
- **Stream ciphers** tend to be faster but require careful key management to maintain security.
- **Block ciphers** require padding when plaintext size is less than block size.
- **Key reuse is dangerous**; security is improved by using **Initialization Vectors (IVs)**.
- IVs combine with the master key to generate unique encryption keys per message.
- IVs are sent in plaintext alongside ciphertext to enable decryption.
- Example: **802.11 wireless encryption frames include IVs in plaintext before encrypted payloads**.
