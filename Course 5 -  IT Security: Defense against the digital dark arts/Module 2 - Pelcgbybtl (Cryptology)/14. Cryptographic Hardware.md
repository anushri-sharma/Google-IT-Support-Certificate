 🔐 **Trusted Platform Module (TPM)** overview  
  – **TPM** is a dedicated crypto processor typically integrated into computer hardware that provides **secure key generation, random number generation, remote attestation, and data binding/sealing**.  
  – TPMs contain a **unique secret RSA key burned into the hardware at manufacture**, enabling hardware-level authentication and detection of unauthorized hardware changes.

📜 Remote attestation  
  – **Remote attestation** lets a system authenticate its software and hardware configuration to a remote party so the remote party can verify platform integrity.  
  – A TPM performs attestation by producing a secure hash of the system configuration signed or protected using the TPM’s **unique hardware-backed key**.

🔒 Data binding vs. data sealing  
  – **Data binding**: derive a unique encryption key from the TPM’s secret and use it to encrypt data, effectively **binding the data to the specific TPM/system** so only that TPM can decrypt.  
  – **Data sealing**: encrypt data with a hardware-backed key but require the TPM to be in a **specified system state** (measured configuration) for decryption, preventing access if the platform has been altered.

🧩 TPM implementations and platform-secure alternatives  
  – TPM is a standard with multiple revisions and can be implemented as a **discrete hardware chip**, integrated on-chip, implemented in firmware/software, or virtualized in a hypervisor.  
  – **Discrete TPM chips** are the most secure implementation because they can include physical tamper resistance.  
  – Mobile devices use a **secure element** (tamper-resistant chip on the SoC or mainboard) for secure key storage and protected app execution.  
  – The **Trusted Execution Environment (TEE)** is an evolution that provides a full isolated execution environment running alongside the main OS, isolating secure applications and processes from the main OS and each other.

⚠️ Trust model and physical-attack considerations  
  – **Manufacturing trust issue**: because the TPM secret is written at manufacture, the manufacturer could potentially access or record keys at that time; keys retained by the manufacturer could enable TPM duplication and undermine expected security guarantees.  
  – **Physical attack evidence**: at least one reported attack used an electron microscope and micron-precision circuitry manipulation to read a TPM’s entire contents; the attack required highly specialized, time-consuming equipment, demonstrating that tamper protections can be bypassed under extreme conditions.  
  – Implication: discrete TPMs raise the bar for attackers via tamper resistance but do not make hardware immune to well-resourced, physical attacks.

💽 Full Disk Encryption (FDE) fundamentals  
  – **FDE** encrypts the entire drive (not just selected files) to protect all disk contents from theft or tampering.  
  – Common implementations: **PGP**, **BitLocker (Microsoft, integrates with TPM)**, **FileVault 2 (Apple)**, **dm-crypt (Linux)**.  
  – Typical layout: one encrypted data partition (typically the root/OS volume) plus a small **unencrypted boot partition** that contains the kernel, bootloader, and **initrd** (initial RAM disk).  
  – Boot process: BIOS/firmware loads the unencrypted boot components, which then prompt for a **passphrase** (or use TPM-protected keys) to unlock the encrypted volume and continue booting.

🔑 TPM integration with FDE and platform integrity enforcement  
  – TPM can store or protect disk encryption keys and enforce platform integrity checks so the disk cannot be unlocked if the system configuration changes.  
  – This integration defends against attacks such as **hardware tampering, disk theft, or disk cloning** by refusing to release encryption keys unless expected measurements match.

🎲 Randomness and entropy in cryptography  
  – A foundational requirement across cryptographic systems is **high-quality randomness** for keys and nonces; weak or predictable randomness (pseudorandomness without sufficient entropy) can create exploitable patterns.  
  – Operating systems maintain an **entropy pool** to seed random number generators; security appliances and servers may include **dedicated hardware RNGs** and vetted PRNGs to ensure cryptographic-quality randomness.  
  – Implication: reliable entropy sources and robust RNG designs are critical to prevent key-recovery attacks and preserve cryptographic strength.
