### Overview of Network Hardware Hardening and DHCP Risks  
- The video begins by introducing **network hardware hardening** techniques relevant to IT support specialists.  
- **DHCP (Dynamic Host Configuration Protocol)** is highlighted as a critical network service that assigns IP configuration to devices. Because of its importance, DHCP is a frequent target for attackers.  
- A major threat discussed is the **rogue DHCP server attack**, where an attacker deploys a fake DHCP server to hand out malicious IP configuration data, such as gateway or DNS server addresses controlled by the attacker. This enables traffic interception and subsequent attacks.  

 
### Mitigation of Rogue DHCP Attacks via DHCP Snooping  
- Enterprise switches implement **DHCP snooping** to counter rogue DHCP servers.  
- DHCP snooping involves:  
  - Monitoring DHCP traffic passing through the switch.  
  - Tracking IP address assignments and mapping them to physical switch ports, creating a binding table.  
- This binding table supports protection not only against rogue DHCP but also **IP spoofing** and **ARP poisoning** attacks.  
- Administrators must configure trusted interfaces:  
  - Trusted DHCP server IP or DHCP helper interfaces.  
  - Trusted uplink ports from where legitimate DHCP responses originate.  
- Any DHCP response from an untrusted port or IP is discarded by the switch, effectively blocking rogue DHCP servers.

### Dynamic ARP Inspection (DAI) to Prevent ARP Spoofing Attacks  
- ARP (Address Resolution Protocol) is vulnerable because it lacks authentication; attackers can send **gratuitous ARP responses** forging a MAC address for a victim's IP.  
- This enables **Layer 2 man-in-the-middle attacks**, redirecting traffic intended for the victim to the attacker’s device.  
- **Dynamic ARP Inspection (DAI)** is a switch-based security feature that:  
  - Uses the DHCP snooping binding table to verify ARP packets.  
  - Drops forged gratuitous ARP packets that do not match known bindings.  
  - Implements **rate-limiting** on ARP packets per port to prevent ARP scanning, which typically precedes ARP spoofing attacks.

### IP Source Guard (IPSG) for IP Spoofing Prevention  
- **IP Source Guard (IPSG)** works alongside DHCP snooping on enterprise switches to block IP spoofing.  
- IPSG dynamically creates filters for each switch port based on the DHCP snooping table.  
- It drops any packet that originates from a port but does not match the assigned IP address for that port.  
- This is particularly useful in small companies using enterprise-grade hardware to enforce strict IP-to-port binding.  


### Introduction to 802.1X Network Access Control  
- **802.1X** is an IEEE standard for port-based network access control using **EAP (Extensible Authentication Protocol)** encapsulation over LAN (EAPoL).  
- Originally designed for Ethernet, 802.1X also supports Wi-Fi and fiber networks.  
- There are many EAP authentication types (~100), but the video focuses specifically on **EAP-TLS**, one of the most common and secure methods.

### The 802.1X Authentication Model: Supplicant, Authenticator, and Authentication Server  
- Three main parties are involved in 802.1X authentication:  
  - **Supplicant:** The client device or software that requests network access and handles authentication (e.g., WPA supplicant on Linux).  
  - **Authenticator:** A network device (typically an enterprise switch or wireless access point) that acts as a gatekeeper, requiring authentication before granting network access.  
  - **Authentication Server:** Usually a **RADIUS server** that performs the actual verification of credentials. The authenticator simply forwards requests to this server.  


### EAP-TLS Authentication Details and Security Benefits  
- **EAP-TLS** uses **TLS (Transport Layer Security)** to perform **mutual authentication** between the client and the authentication server.  
- This method requires clients to present a valid, CA-signed **client certificate** for authentication.  
- Authentication can be:  
  - Certificate-based only.  
  - Certificate plus username/password.  
  - Certificate plus multi-factor authentication such as a one-time password.  
- The security strength of EAP-TLS stems from the robustness of TLS and **Public Key Infrastructure (PKI)**.

### Parallels Between EAP-TLS and HTTPS Security  
- EAP-TLS shares similarities with HTTPS security:  
  - HTTPS combines HTTP with SSL/TLS cryptographic protocols.  
  - TLS optionally uses client certificates for authentication, just as EAP-TLS requires client-side certificates.  
- Proper management of PKI elements is critical for EAP-TLS security:  
  - Protecting private keys.  
  - Distributing CA certificates to clients for server verification.

### Enhancing Security of EAP-TLS with TPM and Full Disk Encryption  
- Security can be further improved by **binding client certificates to client hardware** using **Trusted Platform Modules (TPMs)**, which prevent theft or misuse of certificates.  
- When combined with **Full Disk Encryption (FDE)**, even physical theft of the client device does not compromise network security.  
- These measures strongly reduce the risk of credential theft and unauthorized network access.

### Core Concepts and Terminology  
- **Rogue DHCP Server:** Unauthorized DHCP server handing out malicious network configurations.  
- **Gratuitous ARP:** Unsought ARP response that can poison ARP caches for MITM attacks.  
- **Supplicant:** Client device/software requesting network access using 802.1X.  
- **Authenticator:** Network device enforcing access control, forwarding authentication to server.  
- **Authentication Server:** Backend server validating credentials (typically RADIUS).  
- **EAP-TLS:** An EAP method using TLS for strong mutual authentication via certificates.  
- **TPM:** Hardware module securing cryptographic keys and certificates on client devices.  
- **FDE:** Full Disk Encryption, protecting data at rest on client devices.

- Proper management of cryptographic keys and certificates is critical for maintaining the security benefits of EAP-TLS.
