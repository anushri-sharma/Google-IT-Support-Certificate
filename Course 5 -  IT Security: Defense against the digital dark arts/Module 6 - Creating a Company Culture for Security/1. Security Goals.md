
# [Security Goals](https://www.youtube.com/watch?v=icgZIpAyVv8&t=36s)

🔥 Define security goals before designing architecture  
  – Security design must reflect **company priorities**, data-handling expectations, and **legal/regulatory requirements**.  
  – Example regulatory requirement: **PCI DSS** for organizations handling credit card payments.

💡 Distinguish objectives vs requirements  
  – **Objectives** = end goals (what you want to achieve).  
  – **Requirements** = actions and controls that help achieve those objectives.  
  – Requirements often include more detailed implementation guidance that clarifies ambiguous terms (e.g., what counts as an **open network**).  

🔐 PCI DSS: six high-level objectives (overview)  
  – PCI DSS organizes security into **six objective categories**, each containing specific requirements and implementation guidance.  

🔥 Objective 1 — **Build and maintain a secure network and systems**  
  – Requirements include **installing and maintaining firewall configurations** to protect cardholder data and **not using vendor-supplied defaults** for passwords and security parameters.  
  – Implementation detail example: firewall configurations should **restrict connections between untrusted networks and systems in the cardholder data environment**.  

💳 Objective 2 — **Protect cardholder data**  
  – Requirements: **protect stored cardholder data** and **encrypt transmission of cardholder data across open/public networks**.  
  – Guidance recommends **strong cryptography** and clarifies what constitutes transmission over an **open network**.  
  – Non-technical controls: **data retention policies** — do not retain sensitive payment/authentication data longer than required; **securely delete authentication data once authorization completes**.  

🛡️ Objective 3 — **Maintain a vulnerability management program**  
  – Requirements include **malware protection (anti-virus)** and **secure development/maintenance of systems and applications**.  
  – Implementation specifics: ensure **anti-virus is installed and kept up to date**, run **regular scans and retain logs**, and use **third-party vulnerability databases** to identify known issues.  
  – Patch management expectation: **apply security patches within one month of release**.  

🔑 Objective 4 — **Implement strong access control measures**  
  – Three requirements: **restrict access to cardholder data by business need-to-know**, **identify and authenticate access to system components**, and **restrict physical access to cardholder data**.  
  – Authentication minima: **password authentication** for system access and **two-factor authentication for remote access**.  
  – Physical access controls are emphasized to mitigate **theft and physical compromise** in addition to virtual attacks.  

📈 Objective 5 — **Regularly monitor and test networks**  
  – Requirements: **track and monitor all access to network resources and cardholder data**, and **regularly test security systems and processes**.  
  – Typical controls: **intrusion detection systems (IDS)**, **vulnerability scans**, and routine testing of defenses and alerting to confirm they function as expected.  
  – Testing is essential because **deployment alone does not guarantee effective protection**.  

📜 Objective 6 — **Maintain an information security policy**  
  – Single requirement: maintain a policy addressing information security **for all personnel**.  
  – Policies govern user behavior, reduce accidental circumvention of controls, and must be **easy to find and read**.  
  – Security is **everyone’s responsibility**, not just the security team.  

⚖️ Cross-cutting points, nuances, and implications  
  – Some requirements are **procedural and policy-driven**, not purely technical (e.g., retention and deletion rules).  
  – Detailed requirements clarify ambiguous terms and provide examples, but still require organization-specific interpretation and implementation.  
  – Effective security combines **technical controls, policies, procedures, monitoring, and regular testing**.  
  – User behavior can **weaken security** unintentionally; policies and accessible guidance help mitigate that risk.
