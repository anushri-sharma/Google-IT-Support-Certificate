
# [Multifactor Authentication](https://www.youtube.com/watch?v=tb9lb1Yldro&t=2s)

🔐 **What multi-factor authentication (MFA) is**
  – MFA requires presenting **multiple pieces of evidence** from different categories to authenticate a user.  
  – The three factor categories: **something you know**, **something you have**, **something you are**.  
  – **Effective MFA uses at least two different categories** (e.g., password + token), not multiple items of the same category.

**The many factors that comprise a multi-factor authentication system can be categorized into three types;**

<img width="1279" height="295" alt="image" src="https://github.com/user-attachments/assets/5c4fdfa3-afb8-4f0d-b6a6-fb9775ddedfa" />

🧠 **"Something you know"**
  – Examples: **passwords** or **PINs**.  
  – Multiple passwords alone provide limited extra protection because they remain vulnerable to the same attacks (phishing, keylogging).

🗝️ **"Something you have" — physical tokens and forms**
  – Forms: **USB tokens**, **standalone token generators**, **smartphone apps**, or even **physical keys/locks**.  
  – Common behaviour: generate a **short-lived one-time password (OTP)** that the user enters with username/password.  
  – Example device: **RSA SecurID** (battery-powered display showing rotating OTPs).

⏱️ **Time-based tokens (TOTP) — how they work and risks**
  – Mechanism: device stores a **secret seed**, server stores same seed; OTP = function(seed, current time).  
  – Requires roughly synchronized clocks (commonly via **NTP**).  
  – Security implication: if an attacker obtains the **seed**, they can **clone the token** because time is not secret.  
  – Benefit: user only needs possession or visible display to authenticate; drawback: sync dependency.

<img width="1278" height="600" alt="image" src="https://github.com/user-attachments/assets/7db2e974-dc88-460e-9fd4-d4ca3b4dcf6c" />

🔁 **Counter-based tokens (HOTP or counter tokens) — how they work and why they can be stronger**
  – Mechanism: token uses **secret seed + counter**; counter increments each OTP generation and the server increments on successful auth.  
  – Harder to clone because attacker must recover **both seed and current counter**.  
  – A cloned token becomes **unsynced quickly** as the real token's counter advances, limiting attack window.  
  – Requires server handling for counter resynchronization and tolerance.

📱 **App-based token generators**
  – Smartphone apps implement TOTP/HOTP functionality (same seed-based mechanisms) and remove the need for a separate dedicated device.  
  – Pros: fewer physical devices to carry; cons: still requires support (lost/damaged phone, app transfer, re-seeding).

📨 **SMS-delivered OTPs — common practice and major weaknesses**
  – Widely used method: send OTP via **SMS** to the user's phone.  
  – Weaknesses: **SMS is not encrypted or private**, can be intercepted by well-resourced attackers.  
  – Social-engineering attack: **SIM swap/number porting** or impersonation of the user at the mobile carrier can redirect SMS/calls to the attacker, yielding full account access if password is already compromised.  
  – Implication: SMS-based MFA is less robust against nation-state or targeted attackers and has been criticized for those vulnerabilities.

🎭 **Phishing and man-in-the-middle attacks remain a threat even with OTPs**
  – Attack flow: attacker sends a fake login page, victim enters username, password, **and OTP**; attacker immediately uses those credentials to access the real service.  
  – Result: possession of the OTP at time of capture is sufficient for account takeover in real-time phishing/MITM scenarios.  
  – Conclusion: OTPs mitigate many threats but **do not eliminate phishing** without additional protections (e.g., phishing-resistant protocols).

⚖️ **Usability, operational trade-offs, and support overhead**
  – Physical tokens add **convenience costs**: carrying a device, potential battery failure, loss, damage.  
  – Operational costs: replacement, helpdesk support, device resynchronization, and lifecycle management.  
  – Manual entry friction: users must retrieve device/phone and transcribe OTPs into login pages.  
  – App-based solutions reduce some friction but still require provisioning, backup, and recovery processes.

🔎 **Threat model and cloning considerations**
  – To compromise TOTP: attacker needs the **secret seed** (time is public).  
  – To compromise HOTP/counter tokens: attacker needs **seed + current counter**; clone usefulness decays as counter advances.  
  – SMS compromise requires access to the **phone number or carrier controls** rather than token seed.

✅ **Key security takeaway**
  – **MFA is most effective when factors are from different categories** (knowledge + possession or biometrics).  
  – Be aware of specific weaknesses of each second factor (seed disclosure, SIM swap, phishing/MITM) and plan mitigations (phishing-resistant methods, secure token provisioning, out-of-band verification).
