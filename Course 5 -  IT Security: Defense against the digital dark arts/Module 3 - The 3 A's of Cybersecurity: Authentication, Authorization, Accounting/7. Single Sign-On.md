
# [Single Sign-On](https://www.youtube.com/watch?v=Gfl1XA8DtV0&t=16s)

🔥 **What SSO is**  
  - **Single sign-on (SSO)** lets a user **authenticate once** to gain access to many different services and applications without re-entering credentials for each one.  
  - Authentication is handled by a **central authentication server** (examples: **LDAP**, Kerberos), which issues a **cookie or token** used by SSO-enabled applications to grant access.

💡 **Core mechanism**  
  - A user authenticates to the central service and receives a **session token/cookie**.  
  - Applications configured for SSO accept that token in place of separate username/password input.  
  - Tokens prove the user is authenticated to multiple relying services until the token **expires** or is revoked.

🔑 **Kerberos as an SSO example**  
  - User authenticates once to Kerberos and is granted a **ticket-granting ticket (TGT)**.  
  - The TGT is presented to the **ticket-granting service** to obtain service-specific tickets instead of sending raw credentials to each service.

👍 **Primary benefits**  
  - **One set of credentials** reduces the need for multiple usernames/passwords across services.  
  - Less likelihood of **passwords being written down or stored insecurely**.  
  - Reduces support overhead (fewer password reset requests) and time lost re-authenticating during the workday.  
  - Simplifies access management for users and administrators when many services participate.

⚠️ **Security downsides and risks**  
  - Compromise of an SSO account gives an attacker access to **all services permitted** by that account.  
  - SSO broadens the impact radius: one breach can yield **wide lateral access** across systems.  
  - Attackers can target **session tokens or cookies** (session hijacking) instead of credentials, allowing access without re-authenticating.

🛡️ **Multi-factor authentication (MFA) and its limits**  
  - **MFA is strongly recommended** alongside SSO to mitigate credential compromise.  
  - However, theft of a valid session token/cookie can **bypass MFA** for the duration of the token’s validity because tokens permit access without repeating full authentication until expiry.

🌐 **OpenID example and flow (decentralized SSO)**  
  - **OpenID** is an open decentralized authentication standard where sites (relying parties) delegate authentication to a third-party **identity provider**.  
  - Benefits: sites do not need to implement full authentication infrastructure; users can sign in using an existing identity provider account (no new account creation).  
  - Typical flow:  
    - The **relying party** locates the user’s OpenID provider and **establishes a shared secret** if one is absent.  
    - The shared secret is used to **validate messages** from the provider.  
    - The user is **redirected** or shown a provider login window to authenticate.  
    - After authentication, the user is asked to **confirm trust** for the relying party.  
    - The provider sends an **authentication token** (not the user’s raw credentials) to the relying party indicating the user is authenticated.

🔎 **Important implementation details and implications**  
  - Tokens are the normal artifact of authentication handoff; services receive **tokens, not raw credentials**.  
  - Shared secrets between provider and relying party are used to **verify authenticity** of provider messages.  
  - Token expiry and revocation policies critically affect security posture: **shorter lifetimes and revocation** reduce risk from stolen tokens.  
  - Attack surface shifts: defenders must protect credential stores **and** session/token management infrastructure (cookie security, token storage, transport security).
